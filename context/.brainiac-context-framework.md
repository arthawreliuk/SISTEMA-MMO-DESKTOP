# Brainiac Context Framework

## Framework Overview

**The Problem**: AI-generated code works, but context disappears. Three months later, your own code looks foreign.

**The Solution**: Brainiac Context makes context the primary artifact. Code becomes its manifestation, not the other way around.

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│   INTENT    │ ──► │    BUILD    │ ──► │    LEARN    │
│  What & Why │     │  AI + Human │     │   Update    │
└─────────────┘     └─────────────┘     └─────────────┘
                                               │
                          ◄────────────────────┘
                              Feedback Loop
```

**The 3 Core Steps:**

1. **Intent** - Define what to build and why (create context)
2. **Build** - AI generates code following your context (human supervises)
3. **Learn** - Update context with learnings (refine and improve)

**Why Brainiac Context Works:**

- ✅ **Context is primary** - AI understands your architecture and patterns
- ✅ **Decisions preserve full context** - Every choice documented with rationale
- ✅ **Knowledge evolves with your system** - Living context that stays current
- ✅ **Works with any methodology** - Integrates with Scrum, Kanban, DevOps, or your own process
- ✅ **Simple to adopt** - Just 3 steps, similar to Scrum's simplicity

**Brainiac Context is not a replacement for Scrum or Agile** - it's a complementary framework specifically designed for AI-First development that can be used alongside existing methodologies.

**Customization**: The 3 steps (Intent, Build, Learn) are fixed, but you can adapt how you execute each step to fit your workflow and team needs.

## Framework Philosophy: Agent = Model + Harness

Brainiac Context implements the Karpathy principle: **Agent = Model + Harness**.

The Model (Claude, GPT, etc.) is only as effective as its Harness. Brainiac Context is the structured implementation of that Harness.

**Harness has three sides:**

**Feedforward** — what flows INTO the model before execution:
- `AGENTS.md` (operational router)
- `context/intent/` (product-spec, project-intent, features, bugs, refactors)
- `context/decisions/` (ADRs)
- `context/knowledge/` (patterns, anti-patterns)

**Memory + Bootstrap** — how the model orients itself at session start:
- `/brainiac-context-init` (skill)
- Setup commands in `AGENTS.md`
- `context/evolution/todo.md` (progress)
- Git history

**Sensors (feedback)** — what flows BACK from execution:
- Linters, type checkers
- Tests, E2E
- Review agents
- `context/sensors/sensors.md` (tool configuration + model reaction rules)

These three sides together turn a Model into an Agent that operates within your project's reality.

> **Note on runtime**: Brainiac Context is **design-time discipline**. The same harness can be applied through direct IDE sessions (Claude Code) or layered with an external orchestrator (e.g., [Paperclip](https://github.com/paperclipai/paperclip)) for async multi-agent execution with formal audit log. The orchestrator is optional and replaceable; Brainiac artifacts in the repo remain the source of truth either way. See [`integrations/paperclip/`](integrations/paperclip/) for the orchestrator integration.

## The Brainiac Harness — Required and Optional Files

The Brainiac Harness is materialized in the project as a set of files in `context/`. These files are **not optional documentation** — they are the substrate the framework operates on. Treating them as deletable comments breaks the framework, similar to deleting `/etc/passwd` on Linux.

The Harness is partitioned into three tiers:

### Directory layout — canonicals vs project extensions

Since v1.15.0 the `context/agents/` directory is partitioned into two subfolders that separate framework canonicals from project-specific customization. **Naming note (v1.18+):** the project-owned subfolder is named `project/` (renamed from `extensions/` in v1.15.0–v1.17.x — "extensions" was ambiguous against potential framework-level extensions packs in the future). Legacy projects on `extensions/` are auto-migrated to `project/` by `/brainiac-context-update`.

```
context/agents/
├── brainiac/                  # Framework canonicals (DO NOT EDIT without skill update)
│   ├── agent-developer.md       (Core)
│   ├── agent-reviewer.md        (Core)
│   ├── agent-qa.md              (Core)
│   ├── agent-devops.md          (Extended)
│   ├── agent-planner.md         (Extended)
│   └── agent-insights.md        (Specialist)
└── project/                # Project Extensions (project-specific customization)
    ├── agent-<domain>.md         (domain-specific, e.g., agent-billing.md)
    ├── agent-<named-extension>.md (hybrid named extension, e.g., agent-code-review.md)
    └── ...
```

**Rationale**: previously canonicals and project-specific agents coexisted in a flat `context/agents/`. This forced the update skill to disambiguate by name or heuristic and made it easy for AI without Brainiac context to silently edit canonicals. The two-subfolder split makes the distinction structural — IA distinguishes instantly by path; protection layers (below) target only `brainiac/`.

**Legacy detection**: projects on framework versions < 1.15.0 carry the flat layout. The `/brainiac-context-update` skill detects this and proposes migration as a 🔵 Deterministic finding (no archeology required).

### Core harness (mandatory — always installed)

Without these, central framework flows break:

| File | Why mandatory |
|---|---|
| `context/code-standards.md` | Loaded by `agent-developer.md` in every Build; checklist for `agent-reviewer.md` in Learn Phase 1 |
| `context/agents/brainiac/agent-developer.md` | Persona invoked in every code generation (Build) |
| `context/agents/brainiac/agent-reviewer.md` | Persona invoked in Learn Phase 1 (Quality Gate) — without it, Phase 1 silently breaks |
| `context/agents/brainiac/agent-qa.md` | Persona for validation policy across the project |
| `context/sensors/sensors.md` | Feedback contract — required before Build per Iron Law 3 (Verification Gate) |

Installed automatically by `/brainiac-context-init` and `/brainiac-context-existing`. Cannot be opted out.

### Extended harness (default-on, opt-out at install time)

Useful for most projects. Installed by default; opt-out requires explicit reason at install time:

| File | Opt-out scenario |
|---|---|
| `context/agents/brainiac/agent-devops.md` | Project has no deploy flow (e.g., pure library) |
| `context/agents/brainiac/agent-planner.md` | Project handles intent capture entirely manually, no agent involvement |

Once installed, removing them after the fact requires the same justification as Core harness — they are part of the substrate.

### Specialist harness (opt-in only)

Installed on demand when the project adopts the practice:

| File | Trigger |
|---|---|
| `context/agents/brainiac/agent-insights.md` | Project adopts structured retrospective/insights practice |

### Protection — defense in depth (4 layers, v1.19.1+)

Canonicals under `context/agents/brainiac/` are treated as **substrate**: framework-owned files that should not change except via `/brainiac-context-update`. Four independent layers enforce this — three live inside the framework, and one (Layer 0) lives outside, at the GitHub API boundary:

| Layer | Mechanism | When it fires | Bypass |
|---|---|---|---|
| 0 — **PAT scope** (fine-grained GitHub Personal Access Token) | The AI's token has read+write **scoped only to specific permissions** in **specific repos**. Out-of-scope API calls (delete repo, force-push to main, transfer ownership, modify workflows, access org-level secrets) are rejected by the GitHub API itself. | Any AI action that hits the GitHub API and exceeds the token's granted permissions | None — this is the only enforcement the AI cannot disable locally. Bypass requires the human to grant a broader-scoped token, which is a deliberate authorization decision. See `GETTING_STARTED.md` "Security: auth for AI to act in your PROJECT repo" |
| 1 — **Git pre-commit hook** (`brainiac-canonical-protection.sh`) | Universal git hook (works in any IDE/agent). Blocks commits that modify `context/agents/brainiac/*.md` without an `Allow-Canonical-Edit: <reason>` trailer in the commit message or a `--no-verify`. | Any `git commit` touching `brainiac/` files | `Allow-Canonical-Edit:` trailer (requires explicit reason) or `--no-verify` (last resort) |
| 2 — **Claude Code PreToolUse hook** (`brainiac/canonical-gate.sh`) | Intercepts Edit/Write tool calls targeting `context/agents/brainiac/*`. Returns a clear deny message with the canonical override pattern. | Any Edit/Write tool call inside Claude Code | Comment `<!-- brainiac-canonical-edit: <reason> -->` in the edited region (documented justification) |
| 3 — **Reviewer Phase 1** | `agent-reviewer.md` flags diffs in `context/agents/brainiac/` as `[BRAINIAC-CANONICAL]` finding. 🔴 Block without justification, 🟡 Soft if justified inline. | Learn Phase 1 Quality Gate | Justification must already be in code (referencing skill update or explicit override) |

Layers are independent — bypassing one does not disable the others. A canonical edit reaching Phase 1 without justification gets blocked even if pre-commit was skipped. **Layer 0 is special**: it operates outside the framework's control surface — even if Layers 1-3 are all disabled (hooks deleted, reviewer bypassed), GitHub's API itself still rejects out-of-scope actions. This is why fine-grained PAT scoping with minimum permissions is the **most important security measure** for AI-assisted GitHub operations.

In addition to the gates above, every harness file still carries the marker header that documents the contract:

```html
<!--
Brainiac Harness File — DO NOT DELETE.
Canonical: edit through /brainiac-context-update.
Customization: add agents in context/agents/project/ instead.
-->
```

The marker is informational; the three layers above are the enforcement.

### Project Extensions — how to customize

Customizations live in `context/agents/project/`. Two flavors:

| Flavor | Example | When to use |
|---|---|---|
| **Domain-specific extension** | `agent-billing.md`, `agent-iara-semantic-spam-guard.md` | New capability the canonical layer does not address (project-specific reviewer, stack-specific helper, integration-specific agent) |
| **Hybrid named extension** | `agent-code-review.md` (fulfills the role of `agent-reviewer.md` but with project overrides) | Canonical does not fit and a wholesale custom version is justified — header documents the canonical it replaces |

Every extension file starts with this header (canonical: `templates/project-extension-agent.md.template`):

```html
<!--
Project Extension — Brainiac Harness v<X.Y>
Type: <hybrid-named-extension | domain-specific>
Base canonical: <agent-reviewer.md | none>
Relationship: <one line describing how this relates to a canonical or to project domain>
Created: YYYY-MM-DD
-->
```

This header is consumed by `/brainiac-context-update` and `/brainiac-context-lint` to recognize extensions as first-class.

### Customization vs deletion

| Action | Where | Allowed? |
|---|---|---|
| Add a new agent under `context/agents/project/` (domain or hybrid named) | `project/` | ✅ Yes — primary customization mechanism |
| Edit content of a canonical under `context/agents/brainiac/` | `brainiac/` | ❌ Not by default — requires justified bypass (3 layers above) |
| Edit a Project Extension under `context/agents/project/` | `project/` | ✅ Yes — extensions are project-owned |
| Delete a Core/Extended canonical | `brainiac/` | ❌ No — breaks the framework |
| Move the file out of `context/agents/` | n/a | ❌ No — skills reference the canonical path |

### Update flow

When the framework releases a new version of a canonical agent template, run `/brainiac-context-update`. The skill:

- diffs every file under `context/agents/brainiac/` against canonical templates,
- preserves files under `context/agents/project/` (treated as project-owned),
- proposes deterministic migrations (renames, copies, structural moves) under 🔵 and IA archeology for changes that require inference under 🟡.

Customizations under `project/` don't get touched.

## Techniques vs Artifacts

The framework distinguishes two kinds of moving parts that operate together:

- **Artifacts** are the things written down — `intent/`, `decisions/`, `knowledge/patterns/`, `evolution/buglog.md`, `sensors.md`. Artifacts persist beyond the session and form the living context.
- **Techniques** are the disciplines used while producing or updating artifacts — how to drive the conversation that produces an intent, how to investigate a bug before fixing, how to verify a claim before declaring done. Techniques shape behavior at the moment of execution.

Both are first-class citizens. A skill in `skills/` carries both: it produces an artifact (output) and applies a technique (process). This is what separates Brainiac Context from a directory template.

| Layer | Where it lives | Examples |
|---|---|---|
| Artifacts | `context/` directory + `AGENTS.md` | feature-*.md, ADR, pattern, buglog, sensors.md |
| Techniques | Inside skills (`skills/`) and operational rules in this spec | Plan→Approve→Execute, root-cause-first investigation, verification gate, socratic intent dialogue |

Techniques are codified in the **Iron Laws** section below. Each Iron Law is a non-negotiable discipline that constrains how a step or skill is executed.

## Visual Vocabulary (Color Semantics)

Brainiac skills and agents use colored circles (🔴🟠🟡🟢🔵⚪) as visual shorthand for finding categories, severities, and confidence levels. **The same color carries different meanings in different scales** — this section is the single source of truth that disambiguates them.

### Why colors are reused

Each scale evolved organically around the same emoji palette (red = attention, green = positive, blue = informational, etc.). Renaming all of them to non-overlapping symbols would create more churn than benefit. Instead, **every output that uses a color must reference its scale explicitly** so the reader (human or AI) knows which meaning applies.

### The 4 scales

There are 4 orthogonal scales in the framework:

| # | Scale | Purpose | Used by |
|---|---|---|---|
| 1 | **Severity** | How serious is this finding? Should the build block? | `agent-reviewer.md` (Phase 1 Quality Gate), `agent-templates/agent-reviewer.md`, hooks (clean-code-gate, qa-gate, learn-gate) |
| 2 | **Confidence** | How certain is the reviewer about THIS finding? Separate from severity. | `agent-reviewer.md` (Phase 1), `brainiac-context-init`/`existing` (overrides inference), `brainiac-context-update` (IA archeology) |
| 3 | **Drift category** | What kind of update does this finding require? | `brainiac-context-update` (Plane A: framework drift) |
| 4 | **Audit severity** | How serious is the drift between docs and reality? | `brainiac-context-audit` (Plane B), `brainiac-context-lint` (Plane C) |

### Master table — same color, different meanings per scale

> ⚠️ **The same emoji has different semantics in different scales. Always check the prefix `[Scale:X]` in finding output.**

| Color | (1) Severity | (2) Confidence | (3) Drift category (update) | (4) Audit severity |
|---|---|---|---|---|
| 🔴 | **Block** — hard rule violated, must fix before merge | **low** — frágil; needs human confirmation | **Reconciliação manual** — preserve local; do NOT overwrite | **Critical** — broken Intent↔Implementation traceability |
| 🟠 | — | — | — | **High** — Status lying possibly (AC `[x]` without evidence) |
| 🟡 | **Soft** — quality issue, human decides | **medium** — strong inference, not fully proven | **IA archeology** — needs human review item-by-item | **Medium** — anti-pattern violated in production |
| 🟢 | — | **high** — direct evidence, assertive language allowed | — | **Low** — pattern documented but unused |
| 🔵 | **Info** — suggestion for Learn step | — | **Determinístico** — safe to apply automatically | **Info** — soft Status discrepancy, signal for Learn |
| ⚪ | — | — | **Sem mudança (ignored)** — framework does not touch this | — |

### Conflicts to be aware of

The two most confusing overlaps:

1. **🔴 can mean "act now (severe)" OR "wait, low confidence"** — a reviewer finding tagged `🔴 Block` demands immediate fix; the same `🔴` in confidence context (`🔴 low`) means *do NOT act yet, confirm with human first*. Direction of action is **opposite**.
2. **🟢 can mean "act with confidence" OR "ignore (lowest priority)"** — a `🟢 high` confidence proposal is safe to apply; a `🟢 Low` audit finding is the lowest-priority issue you might skip. Direction of action is **opposite**.

3. **🔵 can mean "informational only (don't act)" OR "apply automatically (safe)"** — a `🔵 Info` reviewer finding feeds Learn step without blocking; a `🔵 Determinístico` update finding is the *first thing* the skill applies automatically without per-item APPROVE. Mental model is **opposite**.

### Convention for finding output

Every finding emitted by a skill or agent MUST prefix its scale name to disambiguate:

```
[Severity:🔴 Block]     src/server/auth.ts:42 — `: any` without justification
[Confidence:🟢 high]    Direct evidence: 3 signals converge

[Drift:🔵 Determinístico] MISSING: .claude/hooks/brainiac/canonical-gate.sh
[Drift:🟡 IA archeology]  CREATE: context/code-standards.md (infer overrides)

[Audit:🔴 Critical]     bug-payment-leak.md without implementation
[Audit:🟢 Low]          pattern api-route-proxy.md without usage
```

Skills that emit consolidated reports (e.g., reviewer Phase 1 with both Severity AND Confidence per finding) MUST emit **both** prefixes:

```
[Severity:🟡 Soft] [Confidence:🟢 high]   src/api/orders.ts:18 — magic number 86400 (1 day in seconds)
[Severity:🔴 Block] [Confidence:🔴 low]   src/lib/cart.ts — possible SEC-3 violation; needs human verification of context
```

### Where each scale is fully documented

The full semantic table for each scale (with examples, formulation rules, and action implications) lives next to where it is used:

| Scale | Canonical definition |
|---|---|
| Severity | [`agent-templates/agent-reviewer.md`](agent-templates/agent-reviewer.md) — section "Severity Levels" |
| Confidence | [`agent-templates/agent-reviewer.md`](agent-templates/agent-reviewer.md) — section "Confidence Levels" (includes formulation rules: assertive vs hedged language per level) |
| Drift category (update) | [`skills/brainiac-context-update/SKILL.md`](skills/brainiac-context-update/SKILL.md) — section "Categorias de findings (Diff Report)" |
| Audit severity | [`skills/brainiac-context-audit/SKILL.md`](skills/brainiac-context-audit/SKILL.md) — top of file (5-row table) |

When in doubt about the meaning of a colored finding in any output, return to this section first.

## The 3 Steps

### Step 1: Intent

**Purpose**: Plan and prepare context before building.

| Aspect           | Details                                                                                                                                                                |
| ---------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **What it does** | Defines what to build and why, creates product spec, feature/bug/refactoring intents, makes technical decisions, creates initial living context                        |
| **When**         | Before building (planning phase)                                                                                                                                       |
| **Outputs**      | Product Spec (`product-spec.md`), Intent files (`feature-*.md`, `bug-*.md` (L2 only), `refactor-*.md`), Decision files (`decisions/*.md`), Initial patterns (optional) |
| **Human Role**   | Lead intent capture, validate intent clarity, approve initial context                                                                                                  |
| **AI Role**      | Assist in structuring intent, suggest context organization, generate context from prompts                                                                              |

**Activities:**

- **Create Product Spec** (recommended): Create `product-spec.md` — consolidated product vision with personas, user journeys, NFRs, KPIs, roadmap, and glossary. This document bridges the gap between project-level vision and individual features, ensuring all AI agents share the same product understanding
- **Define Intent**: Create `project-intent.md`, `feature-*.md`, `bug-*.md`, or `refactor-*.md`
- **Create Decisions** (recommended): Create `decisions/*.md` files for significant technical choices. Best practice: plan decisions before Build for faster implementation
- **Create Initial Context**: AI can help generate context structure
- **Identify Patterns**: For existing projects, identify existing patterns. For new projects, define initial patterns

**Intent Types:**

- **Initial** (quick start): Basic intent, refine as you learn
- **Refined** (pre-planned): Detailed intent after team discussion

Both approaches are valid.

---

### Step 2: Build

**Purpose**: Construction phase - AI builds code with context, human supervises.

| Aspect            | Details                                                                                                    |
| ----------------- | ---------------------------------------------------------------------------------------------------------- |
| **What it does**  | AI generates code using living context (intent, decisions, patterns), human supervises and validates       |
| **When**          | After Intent (or during, if iterative)                                                                     |
| **Prerequisites** | Intent Statement (Required), Technical Decisions/ADR (Required before implementation), Patterns (Optional) |
| **Outputs**       | Implemented code, Decision records (if created/updated), Updated context, New patterns (optional)          |
| **Human Role**    | Supervise AI execution, validate code quality, approve implementation decisions                            |
| **AI Role**       | Explain planning approach, generate code based on context (after approval), suggest solutions              |

**Critical Requirement - ADR Before Implementation**: Before implementing any feature, **verify if a technical decision (ADR) exists**. If no decision exists, **create the decision first** in Step 1 (Intent) or at the start of Step 2 (Build) before proceeding. **Do not start implementation without a documented decision**.

**Activities:**

1. **Verify Decision exists** - Check if technical decision exists. If not, create it first
2. **Plan before building** - Load context files, verify ADR is available, ask AI to explain what it will build
3. **Approve before executing** - Review and approve plan, ensure ADR is in place
4. **Execute with context** - AI generates code based on approved plan and documented decisions
5. **Follow patterns** - Use established patterns, avoid known anti-patterns
6. **Run Sensors before declaring done** - Execute commands listed in `context/sensors/sensors.md` and react per their reaction rules. No Build completes without sensors passing (or explicit justification if a sensor is not feasible in this environment)
7. **Pass through Learn Gate** - Build is not complete until Step 3 (Learn) runs. See "Learn Gate" below.

**Updating Existing Features:**

When updating an existing feature (not creating new):

- **Analyze existing code first** - AI should understand what's already implemented
- **Make incremental changes only** - Modify only what needs to change based on updated intent
- **Preserve existing code** - Don't regenerate code that doesn't need to change
- **Reference "Changes from Original"** - If present in the intent file, follow it explicitly
- **Update related files** - Update tests, documentation, and related code as needed

**Example for AI when updating:**

```
Update the existing feature following @context/intent/feature-[name].md.

IMPORTANT: This is an UPDATE to existing code, not a new implementation.
- First, analyze the existing code for this feature
- Identify what needs to change based on the updated intent
- Make ONLY the necessary modifications
- Preserve existing code that doesn't need to change
```

**Writing Prompts for AI Code Generation:**

Since **context is the primary artifact**, prompts should be **simple and reference the context**.

| Approach                            | When to Use                                           | Example                                                                   |
| ----------------------------------- | ----------------------------------------------------- | ------------------------------------------------------------------------- |
| **✅ Simple Prompts** (Recommended) | Context is sufficient                                 | `Implement authentication following @context/intent/feature-user-auth.md` |
| **✅ AI Agents** (Advanced)         | Need structured/reusable execution, working with team | `Execute @context/agents/agent-backend.md for payment feature`            |
| **⚠️ Detailed Prompts** (Avoid)     | Temporary testing/learning only                       | Duplicates context, hard to maintain                                      |

**Recommendation**: Start with Simple Prompts (default). Add AI Agents when you need structure or reusability. Avoid Detailed Prompts - use only temporarily, or create an agent file instead.

### Step 3: Learn

**Purpose**: Learning phase - Update living context to reflect code changes and classify insights into durable artifacts.

| Aspect           | Details                                                                                                                                                                                      |
| ---------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **What it does** | Updates context to reflect actual code changes, classifies insights into patterns/anti-patterns/decision outcomes, refines intent if needed                                                  |
| **When**         | After Build (or continuously during Build)                                                                                                                                                   |
| **Outputs**      | Updated context, Updated decision records (with outcomes), Updated buglog (L1, every bugfix), Updated changelog (L3, only milestones), Preserved/updated patterns and anti-patterns         |
| **Human Role**   | Review code changes, decide what needs context updates, validate context accuracy                                                                                                            |
| **AI Role**      | Explain what changed before updating context, help identify changes, suggest what needs updating, assist in updating documentation                                                           |

**Activities:**

1. **Update Context** (Primary): Plan (ask AI to identify what changed), Approve (review AI's analysis), Execute (update context with approval)
2. **Bug Logging & Changelogs**: Update `buglog.md` after every bugfix. Update `changelog.md` only for milestones.
3. **Classify Insights (No Limbo Rule)**: Every insight from Build must land in one of these durable artifacts — never a loose "learning" file:
   - **Reusable success** → create/update `knowledge/patterns/[name].md`
   - **What to avoid** → create/update `knowledge/anti-patterns/[name].md`
   - **Revisiting a decision** → update `Outcomes` section of the related ADR in `decisions/`
   - **Milestone-worthy event** → entry in `evolution/changelog.md` (L3)
   - **Team process/retro insight** → outside the framework (retrospective, wiki, etc.)
4. **Update Todo List** (If exists): Mark completed items in `evolution/todo.md`, add new tasks discovered during Build
5. **Update Intent Files** (Required after Build):
   - Mark completed Acceptance Criteria as `[x]` in the relevant `feature-*.md` / `bug-*.md`
   - Update the `Status` section: set `Updated` date with current phase and reason
   - Add new `Related` links if new decisions or features were created during Build
   - If ALL Acceptance Criteria are `[x]`, change Status from `In Progress` to `Completed`

**Feedback Loop**: Learnings from Step 3 feed back to Step 1 (Intent) for refinement in the next cycle.

### Learn Gate

The **Learn Gate** is the mandatory checkpoint between Build completion and task closure. No task of Build, bugfix, refactor, migration, workflow change, prompt change, test change, or any behavioral change is considered complete until the Learn Gate is passed.

**Passing the Learn Gate requires one of two outcomes:**

1. **Context updated** — the relevant artifacts in `context/` were touched (intent status, ADR outcomes, buglog, patterns, anti-patterns, changelog) AND the update is documented.

2. **Explicit skip declaration** — if no update is due (e.g., planning-only session, read-only analysis, no implementation), this must be **declared explicitly at task closure with a reason**. Silent skipping is a framework violation.

**Exceptions:** Planning, review, reading, or analysis sessions without implementation do not require `context/` updates by default — but the Learn Gate still runs and the "no update needed, reason: [read-only session]" declaration is required.

**Code changed without context updated does not mean the task is complete.**

## Iron Laws — Operational Discipline

Iron Laws are non-negotiable disciplines that govern how steps and skills are executed. They sit alongside the 5 Principles (philosophy) and the Plan → Approve → Execute contract (collaboration), and they activate the Validation by Layer obligation into concrete behavior.

An Iron Law is broken only by **explicit, declared exception** — never silently. Exceptions are recorded in the relevant artifact (intent file, ADR outcome, buglog, or changelog) with reason and residual risk.

### Iron Law 1 — Failing Test Before Production Code (Build)

No production code is committed without a failing test that exercises the new behavior. The cycle is RED-GREEN-REFACTOR:

1. **RED** — write the test, watch it fail with a clear failure message.
2. **GREEN** — write the minimum code to make the test pass.
3. **REFACTOR** — improve structure without changing behavior.

The test type is chosen per Validation by Layer (unit, integration, e2e, contract, etc.). The discipline is the same regardless of layer.

**When to declare exception:** spike, throwaway prototype, or infrastructure-only change with no behavior. Declared in the intent file with reason.

### Iron Law 2 — Root Cause Before Fix (Bugfix)

No fix is proposed without root cause investigation completed first. The investigation passes through:

1. **Read the error** — full stack trace, exact failure message, line numbers.
2. **Reproduce reliably** — exact steps; if not reproducible, gather more data, do not guess.
3. **Trace data flow** — where does the bad value originate, what called it, where does it propagate.
4. **Form a single hypothesis** — "I think X is the cause because Y", written explicitly.
5. **Test the hypothesis with the smallest possible change.**

If three fix attempts fail, stop and question the architecture — the symptom is wrong; the pattern itself is the problem.

**When to declare exception:** trivial typo or copy-paste fix where root cause is self-evident from the diff. Still logged in `buglog.md`.

### Iron Law 3 — Output Evidence Before Completion Claim (Verification)

No task is declared complete based on memory or expectation. Completion requires fresh output from the relevant verification command:

1. **Identify the verification command** — what produces the evidence (e.g., sensor command from `sensors.md`).
2. **Execute it now** — not "from earlier in the session", not "should pass".
3. **Read the output** — actual lines, actual exit code.
4. **Match against the claim** — "tests pass" requires `0 failures` in this run, not a previous one.
5. **Only then assert completion.**

This Iron Law operationalizes the Learn Gate: passing requires evidence, not assertion.

**When to declare exception:** read-only or analysis sessions with no implementation (already covered by the Learn Gate skip declaration).

### Iron Law 4 — Trade-Off Matrix Before Decision Records (Decision)

No ADR is written without a trade-off matrix that scores at least one alternative against the chosen option. The matrix evaluates against the project's stated constraints (cost, complexity, risk, time-to-deliver, reversibility).

The matrix lives in the ADR's `Alternatives` section. "We considered X but rejected it" without reasoning fails this Iron Law.

**When to declare exception:** decision is forced by external constraint (vendor lock-in, regulatory requirement) — declared in ADR's `Context` section with the constraint cited.

### Iron Law 5 — TDD Applied to Skills (Skill Authoring)

No skill is written without applying the same red-green-refactor discipline to itself:

1. **RED** — describe the workflow the skill is meant to drive without the skill loaded; identify what fails or is missing.
2. **GREEN** — write the minimum `SKILL.md` that closes the gap.
3. **REFACTOR** — close loopholes (rationalizations, escape hatches, ambiguity).

Applies to skills in `skills/`, agents in `agent-templates/`, and project-level agents in `context/agents/`.

### Iron Law 6 — No Accumulated Scope (Small-Steps Discipline)

Every cycle Intent → Build → Learn closes before the next one starts. **No accumulation.**

When the user dumps multiple ideas in a single message (common with beginners, non-technical users, or excited developers who want to "do it all at once"), the agent must:

1. **Process and understand** the full intent (do not interrupt at the first opportunity).
2. **Organize visually** what was detected — list candidate features, candidate bugs, candidate ADRs, separated by category.
3. **Request explicit OK** before any execution: *"Detected N features and M bugs. Confirm this separation? Which do you want to tackle first?"*
4. **Break into GitHub Issues + PRs** — 1 issue per feature/bug, 1 PR per issue (`brainiac-context-github` + `brainiac-context-feature`/`bugfix`).
5. **Use GitHub Project Board** for visual tracking (Model A — see "GitHub as Operational Layer" section).
6. **Close 1 cycle at a time** — finish issue → branch → Draft PR → Sensors → Learn Gate → Merge BEFORE starting the next.

**Why this matters in AI-First development**:

- Context window fills with noise when ideas mix in one execution
- PRs that mix scope (3 features in 800 lines) are unreviewable
- Build that mixes scope drifts from initial intent (vague) into speculative code
- Learn step loses signal when there are 5 changes to capture at once

Full operational pattern lives in `## Small-Steps Discipline` section below (see "Working with Context Files" → "Small-Steps Discipline").

### Iron Law 7 — Context Hygiene Between Cycles

Brainiac must not rely on stale chat history. The context window is a working set, not long-term memory.

1. **Framework spec is always-on.** `CLAUDE.md` imports `AGENTS.md`; `AGENTS.md` imports `context/.brainiac-context-framework.md`. The framework spec is the authoritative source of Brainiac rules and is re-injected from disk on every `/clear` and `/compact` via the import chain.

2. **After `/clear`**, reload Brainiac base before first action: framework spec (automatic via @import chain), AGENTS.md, project intent, code/security standards, sensors, and active task artifacts when applicable.

3. **After `/compact`**, do not rely on the compact summary alone for critical work. Before Build, Learn, framework update, migration, refactor, or complex bugfix, reload active intent and relevant ADRs from disk.

4. **Scope switch prefers `/clear`. Never start a new scope at or above 40%.** A new feature, bug, refactor, migration, or framework update must not start on accumulated context.

5. **Context budget discipline:**
   - 0–30%: healthy
   - 30–40%: continue same scope only
   - 40%+: `/compact` or `/clear` before new scope; never start new scope at or above 40%
   - 50%+: `/compact` or `/clear` before any critical Brainiac workflow (Build / Learn / update / refactor / complex bugfix)
   - 60%+: continuing critical work without `/compact` or `/clear` is a Brainiac violation
   - 75%+: work must stop until `/compact` or `/clear` is run

6. **Hooks may inform, pause, or request user action.** Hooks must not silently run `/clear` or `/compact`. Percentage thresholds are **discipline + advisory** (via `/context` checks and PreCompact hook); they are not enforced as automatic blocks by hooks. The only technical enforcement is scope-switch detection in `UserPromptSubmit`, which surfaces advisories but does not block prompt processing.

7. **Bypass requires declaration** in the active intent, buglog, or ADR with reason and residual risk. Silent bypass is a framework violation.

Full operational details — Clear/Compact Protocol, Context Budget Thresholds table, Critical Work Definition — live in `## Context Hygiene` section below.

### Bypass Protocol

Any Iron Law may be bypassed only by explicit declaration in the relevant artifact:

- The reason for the bypass.
- The residual risk accepted.
- The compensating verification (if any).

Silent bypass is a framework violation, equivalent to silent skipping of the Learn Gate.

## Context Hygiene

Operational details for Iron Law 7. The framework spec is always-on via the import chain `CLAUDE.md → @AGENTS.md → @context/.brainiac-context-framework.md`. This section codifies how to manage the context window between cycles.

### Clear/Compact Protocol

**`/clear` (full reset):**
- All conversation history is dropped.
- `CLAUDE.md` is re-read from disk; the `@AGENTS.md` and `@context/.brainiac-context-framework.md` imports are re-parsed automatically (Claude Code documented behavior).
- **Action required by the agent:** before the first action after `/clear`, confirm framework + intent + standards are loaded. The framework spec is automatic via @import; the active intent file must be re-read explicitly if continuing prior work.

**`/compact` (summarization, manual or auto at 75%):**
- Conversation history is compressed into a summary; some detail is lost.
- `CLAUDE.md` and `@imports` survive (re-parsed from disk).
- Skill descriptions not invoked during the session are dropped.
- **Action required by the agent:** for routine continuation of the same scope, the post-compact summary is sufficient. For critical work (Build, Learn, framework update, migration, refactor, complex bugfix), re-read active intent + relevant ADRs from disk before proceeding.

**Scope switch prefers `/clear`:** when changing from one feature/bug/refactor to another, prefer `/clear` over `/compact` because `/compact` preserves the same-task summary (now noise) while `/clear` resets cleanly. Iron Law 7.4 codifies the 40% threshold above which a new scope must not start.

**Hooks (advisory, not enforcement):**
- `pre-compact.sh` saves a handoff snapshot to `.brainiac/last-compact-state.json` and emits a JSON advisory.
- `post-compact.sh` injects a mandatory reload checklist via JSON `additionalContext`.
- `session-start.sh` reacts to `source` (startup/resume/clear/compact) with differentiated reminders.
- `prompt-reminder.sh` detects scope-switch patterns in `UserPromptSubmit` and emits a soft advisory.
- **No hook silently runs `/clear` or `/compact`.** Context lifecycle commands are human-triggered.

### Context Budget Thresholds

| %      | 1M (Opus 4.7 / Sonnet 4.6) | 200K (default Opus)  | Action                                              |
|--------|----------------------------|----------------------|-----------------------------------------------------|
| 0–30   | healthy                    | healthy              | proceed                                             |
| 30–40  | caution                    | caution              | continue same scope only                            |
| 40+    | compact/clear before scope | compact/clear        | never start new scope at or above 40%               |
| 50+    | compact/clear before crit. | compact/clear        | before any critical Brainiac workflow               |
| 60+    | violation if continued     | violation            | continuing critical work without compact/clear = violation |
| 75+    | stop                       | stop                 | work must stop until `/compact` or `/clear` is run  |

**AI discipline note:** the agent should run `/context` before any critical workflow, parse tokens used, compute `%` against the session's model window (Opus 4.7 = 1M, Sonnet 4.6 = 1M, default Opus = 200K), and apply the table above. This is the agent's responsibility, not hook automation — Claude Code does not expose `context_window.used_percentage` programmatically (as of this framework version).

### Critical Work Definition

A workflow is **critical** (and thus subject to thresholds 50%/60%/75% above) if it falls into any of:

- **Build** (`/brainiac-context-build` or implementation following an intent)
- **Learn** (`/brainiac-context-learn` Phase 1 Quality Gate or Phase 2 Knowledge Capture)
- **Framework update** (`/brainiac-context-update` framework-drift apply)
- **Migration** (data migration, schema change, refactor across modules)
- **Refactor** (`/brainiac-context-refactor` or any change touching ≥3 files with structural impact)
- **Complex bugfix** (L2 or L3 — root cause non-obvious, cross-layer, or generating new pattern/anti-pattern)

Non-critical work (read-only exploration, trivial typo fixes, single-line edits, doc-only changes) is not subject to the critical thresholds; only the 40% new-scope rule applies.

### Operationalization (since v1.21.0)

Iron Law 7 is a rule of discipline. Until Anthropic exposes `context_window.used_percentage` to hooks/agents (B39 parking), the agent applies a **signal-based heuristic** to know when to proactively propose `/compact` or `/clear`.

**Where the operationalization lives:**

- **`agent-templates/agent-developer.md`** and **`agent-templates/agent-planner.md`** Phase 0 — agents apply the heuristic at the start of any workflow they own (Build, Plan, ADR creation, refactor planning)
- **Skills** `/brainiac-context-build`, `/brainiac-context-feature`, `/brainiac-context-bugfix`, `/brainiac-context-decision`, `/brainiac-context-learn` — bootstrap section invokes Phase 0 of the relevant agent before executing the workflow

**Heuristic signals (agent decides advisory is needed if any fire):**

- **Session length** — more than ~30 user turns since session start, OR more than ~5 large reads/writes in a row, OR session crosses multiple feature/bugfix/refactor cycles
- **Workflow boundary** — starting Build/Learn/decision skill, starting new feature/bugfix intent, large refactor about to begin
- **Explicit user signal** — user mentions context bloating, asks if compact is recommended, or shows the context indicator

**Advisory format (agent → user):**

> "Antes de prosseguir com `<workflow>`, recomendo verificar o uso de contexto. Iron Law 7 §5 sugere `/compact` ou `/clear` a partir de 50% antes de workflow crítico. Pode rodar `/context` e confirmar se estamos abaixo do threshold, ou prefere fazer `/compact` direto?"

**User override:**

The user can override with an explicit reason ("ciente, prossiga"); the agent records the override but does not silence the advisory. The override is a deliberate authorization, not an excuse to skip the check.

**This is discipline + advisory, not enforcement.** Real enforcement comes via B39 when the API exposes percentage to hooks.

**Why a heuristic instead of a hard rule:**

Without API access, the agent can't measure context%. A hard rule "always propose `/compact` before every workflow" would be noise. A pure signal heuristic catches the real cases (long sessions, workflow boundaries, user concern) while staying out of the way in short healthy sessions.

## Context Loading Order

When AI starts or resumes work on a project, it must load context in this order (top = first):

1. **`AGENTS.md`** (project root) — operational router
2. **`context/.brainiac-context-framework.md`** — framework spec (if project has a local copy)
3. **`context/intent/product-spec.md`** (if exists)
4. **`context/intent/project-intent.md`** — project scope and tech stack
5. **`context/intent/feature-*.md` / `bug-*.md` / `refactor-*.md`** — only the one relevant to the current task
6. **`context/decisions/*.md`** — only the ones referenced by the loaded intent
7. **`context/knowledge/patterns/*.md` and `anti-patterns/*.md`** — relevant to the layer being changed
8. **`context/code-standards.md`** — Clean Code AI-First standards (project snapshot — see [Code Standards as Build Contract](#code-standards-as-build-contract))
9. **`context/security-standards.md`** — Security gates (project snapshot — see [Security Standards as Separate Layer](#security-standards-as-separate-layer))
10. **`context/agents/agent-*.md`** — the specialist agent for this layer (if exists)
11. **`context/sensors/sensors.md`** — must be loaded before starting Build

**Why order matters**: the framework defines rules (steps 1–2), the project defines scope (steps 3–4), the task defines focus (steps 5–10), and sensors define the feedback contract (step 11). Loading out of order creates gaps.

**Minimum for Build**: framework + project-intent + relevant intent + relevant ADR + code-standards + security-standards + sensors.

## Working with Context Files

### When to Create vs Update

| Situation                | Action                                | Why                                |
| ------------------------ | ------------------------------------- | ---------------------------------- |
| New feature              | Create `feature-*.md`                 | New scope                          |
| Update existing feature  | Update `feature-*.md`                 | Same scope (Git preserves history) |
| New decision             | Create `decisions/003-*.md`           | New scope                          |
| Add outcomes to decision | Update `decisions/002-*.md`           | Same scope                         |
| New pattern discovered   | Create `patterns/new-pattern.md`      | New scope                          |
| Refine pattern           | Update `patterns/existing-pattern.md` | Same scope                         |

**Rule of Thumb**: Ask yourself "Is this a new scope or evolution of existing scope?"

- **New scope** → Create new file
- **Same scope** → Update file (Git preserves history)

**Deprecating (NOT Removing)**:

- When a feature is removed → Mark as deprecated, do NOT delete
- When a bug is resolved:
  - If `bug-*.md` exists (L2) → Mark as `Resolved`, do NOT delete
  - If no `bug-*.md` exists (L1 only) → The record stays only in `buglog.md`
- Keep files for history and traceability

### When to Update vs Create New File (Feature Evolution)

**Key Principle**: Use Git for versioning. Update the same file when it's the same feature evolving. Only create a new file when it's a completely different feature.

| Scenario                         | Action                                      | Example                                                                     |
| -------------------------------- | ------------------------------------------- | --------------------------------------------------------------------------- |
| **Feature refinement**           | Update `feature-*.md`                       | Adding new fields to user auth, changing UI, adding validation              |
| **Feature scope expansion**      | Update `feature-*.md`                       | Adding OAuth to existing auth feature                                       |
| **Technical approach change**    | Update `feature-*.md` + Create new decision | Switching from JWT to session-based auth (same feature, different approach) |
| **Complete feature replacement** | Create new `feature-*.md` + Deprecate old   | Replacing old auth system with completely new one (different architecture)  |
| **Feature split**                | Create new `feature-*.md` files             | Splitting "user management" into "user-auth.md" and "user-profile.md"       |

**How AI Should Handle Feature Updates:**

When you update a feature and execute the agent/prompt, the AI should:

1. **Analyze existing code first** - Understand what's already implemented
2. **Identify what needs to change** - Compare updated intent with current code
3. **Make incremental changes only** - Modify only what's necessary
4. **Preserve existing code** - Don't regenerate code that doesn't need to change
5. **Follow "Changes from Original" section** - If present in the intent file

**Example prompt for AI:**

```
Update the existing feature following @context/intent/feature-[name].md.

IMPORTANT: This is an UPDATE to existing code, not a new implementation.
- First, analyze the existing code for this feature
- Identify what needs to change based on the updated intent
- Make ONLY the necessary modifications
- Preserve existing code that doesn't need to change
```

**How to Document Significant Changes in the Same File:**

When updating a feature file with significant changes, add a "Change History" section:

```markdown
## Change History

### 2024-03-15 - OAuth Integration Added

- **What changed**: Added OAuth (Google, GitHub) to existing email/password auth
- **Why**: User request for social login
- **Impact**: New decision created (005-oauth-integration.md)
- **Status**: Completed

### 2024-02-10 - MFA Added

- **What changed**: Added multi-factor authentication
- **Why**: Security requirement
- **Impact**: Updated decision 002-auth.md with MFA approach
- **Status**: Completed
```

**When to Create a New Feature File:**

Only create a new `feature-*.md` file when:

- It's a **completely different feature** (not an evolution)
- The feature is being **replaced entirely** (then deprecate the old one)
- The feature is being **split into multiple features** (create new files for each)

**Example - When to Update vs Create:**

```
✅ Update feature-user-auth.md:
- Add password reset
- Add email verification
- Add OAuth providers
- Change from JWT to sessions
- Add MFA

❌ Create new file:
- Replacing auth with completely new system → feature-auth-v2.md (deprecate old)
- Splitting auth into separate features → feature-login.md + feature-registration.md
```

**Remember**: Git preserves all history. Updating the same file maintains traceability while keeping context organized. Use Git history to see what changed and when.

### File Naming Conventions

| File Type      | Naming Pattern                               | Example                                                  |
| -------------- | -------------------------------------------- | -------------------------------------------------------- |
| Product Spec   | `product-spec.md`                            | `product-spec.md`                                        |
| Project Intent | `project-intent.md`                          | `project-intent.md`                                      |

### Status and Dates

Include at the end of every context file:

```markdown
## Status

- **Created**: YYYY-MM-DD HH:MM (Phase: Intent/Build/Learn)
- **Status**: [Status Type]
- **Updated**: YYYY-MM-DD HH:MM (Phase: Intent/Build/Learn) - [reason] (optional)
```

**Status Types:**

**Best Practices**: Always include created date, update status when it changes, include phase (which Brainiac Context step), be consistent across all files.

## Small-Steps Discipline

> Operational implementation of [Iron Law 6 — No Accumulated Scope](#iron-law-6--no-accumulated-scope-small-steps-discipline). This section is **referenced by** skills like `brainiac-context-feature`, `brainiac-context-bugfix`, `brainiac-context-decision`, and `brainiac-context-init` — written once here, applied uniformly.

### Why this exists

AI-First development has a failure mode that's invisible until it bites: **scope accumulation**. The user types one paragraph describing 4 features + 2 bugs + 1 architectural concern, and the agent (eager to please) starts implementing everything at once. Output:

- A PR with 800 lines mixing 3 features that nobody can review properly
- Intent files that are vague ("system with chat + cart + auth + billing + analytics")
- Build that drifts from initial intent into speculative code (because no checkpoint forced clarification)
- Context window full of noise; agent forgets the first request by the time it gets to the fourth
- Learn step that captures nothing meaningful because too much changed

This pattern is especially common with:

- **Non-technical users** who describe outcomes ("I want a system that does X, Y, Z") without considering implementation scope
- **Beginners** who don't know how to decompose
- **Excited developers** dumping all ideas at once instead of one cycle at a time
- **Time-pressured users** who skip Plan → Approve → Execute thinking it's slower (it's not — accumulation is slower because rework dominates)

The framework actively prevents this. Agent is **required** to apply the Socratic pattern below whenever the input is non-trivially multi-scope.

### The Socratic pattern (6 steps)

When user input contains more than one scope unit (feature, bug, decision, refactor), the agent must:

**Step 1 — Process and understand fully**

Read the entire input. Do not interrupt at first opportunity. Do not start asking clarification questions on the first paragraph — let the user finish their thought.

**Step 2 — Organize visually**

Present back what was detected, **categorized**:

```
Detectei o seguinte na tua descrição:

📋 Features candidatas:
1. <feature 1> — escopo: <breve>
2. <feature 2> — escopo: <breve>

🐛 Bugs candidatos:
1. <bug 1> — sintoma: <breve>

🏛️ Decisões técnicas candidatas (ADR):
1. <decision 1> — domínio: <breve>

❓ Pontos vagos / a esclarecer:
1. <ambiguity 1>
2. <ambiguity 2>
```

This **single visual organization** is the highest-leverage step. It forces the user to confirm separation, identifies ambiguities, and prevents accumulation downstream.

**Step 3 — Request explicit OK before any execution**

> "Confirma essa separação? Quer ajustar algum item? E qual queres atacar primeiro?"

Wait for the answer. **Never start implementation based on visual organization alone** — that's accumulation under disguise.

**Step 4 — Break into GitHub Issues + PRs (if GitHub mode active)**

Once user confirms the decomposition and priority order, the agent:

1. Creates 1 GitHub Issue per scope unit via `brainiac-context-feature` or `brainiac-context-bugfix` (each with labels, intent path, scope)
2. Each issue goes to the Project Board (Triage column) — see Model A in "GitHub as Operational Layer"
3. The **first** issue selected for execution moves to **Ready** → **In Progress**
4. Other issues stay in Triage / Ready until the current cycle closes

**Step 5 — Track visually via GitHub Project Board (Model A)**

Board state at any moment shows: 1 in progress, N waiting in Ready, others in Triage. User and agent can see the queue. Speculative items can stay in `todo.md` (local) without bloating the Board.

**Step 6 — Close 1 cycle before the next**

The current cycle (issue → branch → Draft PR → Sensors → Learn Gate → merge) **must finish** before the agent picks up the next issue. No parallel implementation of multiple features. No "let me just also do X while I'm here."

If user pushes for parallelism: agent flags it as Iron Law 6 violation and asks for explicit Bypass Protocol declaration (reason, risk, mitigation — see Bypass Protocol section).

### When this applies (and when it doesn't)

**Apply (mandatory)** when input contains:

- 2+ features described in same message
- Feature + bug mixed (separate scope cycles)
- Feature + architectural decision mixed (decision needs ADR before feature)
- Vague high-level description ("make it better", "add a dashboard", "integrate with X") that requires clarification before scope can be defined

**Skip (overkill)** when input is:

- A single concrete bug with clear reproduction steps and 1-file fix
- A typo or trivial doc fix
- A continuation of an active cycle (user said "next step" or "continue") and the current scope is unfinished
- Pure question / discussion (not asking for implementation)

When in doubt, **apply**. The cost of one extra confirmation is minutes; the cost of accumulation is hours of rework.

### Two user modes (configurable in `brainiac-context-init`)

Different users need different levels of guidance:

| Mode | Behavior | Best for |
|---|---|---|
| **`guided`** (default for new projects) | Full Socratic pattern always. Agent asks one question at a time. Confirms before each step. Explains rationale. | Non-technical users, beginners, first-time framework users, complex multi-scope projects |
| **`expert`** | Socratic pattern only when input is multi-scope OR ambiguous. Skips when input is clear single-scope. Less hand-holding. | Experienced devs comfortable with Brainiac, time-pressured iteration, repeated similar tasks |

Mode is set at init time:

```bash
# Asked during /brainiac-context-init:
> "Modo de interação preferido: guided (uma pergunta por vez, mais explicação) ou expert (rápido, assume disciplina)? [guided/expert]"
```

Stored in `context/.brainiac-mode` or in `AGENTS.md` header. Skills load it on Bootstrap and adjust behavior accordingly.

**Switching modes**: edit the file or run `/brainiac-context-init --mode <new>` to update. Safe to switch anytime.

### Integration with other framework mechanisms

| Mechanism | How it reinforces Small-Steps |
|---|---|
| **Iron Law 1 (Failing Test Before)** | Test forces concrete acceptance criterion per cycle — implicit decomposition |
| **Iron Law 2 (Root Cause Before Fix)** | 4 phases enforce sequential investigation — cannot skip to fix |
| **Iron Law 3 (Output Evidence Before)** | Evidence requirement makes Build closure tangible — knows when 1 cycle is "done" |
| **Iron Law 4 (Trade-Off Matrix Before ADR)** | Forces decision moment, separated from implementation |
| **Iron Law 7 (Context Hygiene)** | `/clear` between cycles prevents context bleed; 40% threshold prevents new scope on accumulated session |
| **GitHub Operational Protocol** | Issues + PRs make decomposition externally visible and trackable |
| **`brainiac-context-learn` (Phase 1 + Phase 2)** | Each cycle has its own Learn Gate — accumulation prevented at exit too |

## Bug Logging Policy (3 Levels)

Brainiac Context uses a 3-level standard for bug tracking and documentation, ensuring critical knowledge is captured without overloading routine documentation.

> `buglog.md` is updated in Step 3 (Learn) after every bugfix, including quick-fix mode.

**Level 1 — Buglog (always)**

- Every bug fix must generate an entry in `context/evolution/buglog.md`.
- Minimum entry: Date, Where, Symptom, Cause, Fix, Files changed, Validation/How tested, Links (optional).

**Level 2 — Bug Intent (conditional)**
Create a `context/intent/bug-<slug>.md` file only when the bug has "knowledge value" OR "coordination value".
If it meets **at least 1** of the criteria below, the bug escalates to Level 2:

1. Non-obvious root cause (e.g., race conditions, SSR/hydration, listeners, complex state).
2. Touches a pattern/architecture (shared hooks, RLS, contracts, idempotency, integrations).
3. High regression risk (responsiveness, auth, checkout, customer service).
4. Significant user/operator impact (flow breaking).
5. Required or requires specific testing (Playwright/payload contract/RLS regression).
6. Cross-layer (involves more than one layer).
7. Creates/updates a pattern or anti-pattern in `context/knowledge/...`.

_Note: If the bug does not meet any of the criteria above, log the fix only in Buglog (Level 1)._

**Level 3 — Changelog (milestones only)**

- `context/evolution/changelog.md` is strictly reserved for milestones: releases, significant feature changes, decisions/ADRs with broad impact.
- Do not log routine bugfixes in the changelog.
- A bug only enters the changelog if it was a major incident with high impact, caused an architectural change/ADR, or is part of a bundled "bugfix release".

## Product Spec, Project Intent, Features, Bugs, and Refactoring

Understanding the hierarchy and when to use each file:

**Context Hierarchy (Source of Truth):**

```
Product Spec > Project Intent > Features > Decisions > Code > Tasks
```

| File Type             | When to Use                                                                         | When to Update                                                                |
| --------------------- | ----------------------------------------------------------------------------------- | ----------------------------------------------------------------------------- |
| **product-spec.md**   | Consolidated product vision: personas, user journeys, NFRs, KPIs, roadmap, glossary | Personas change, new journeys discovered, NFRs revised, roadmap reprioritized |
| **project-intent.md** | Project scope, constraints, acceptance criteria, tech overview                      | Project scope changes significantly, high-level goals change                  |
| **feature-\*.md**     | Individual feature requirements                                                     | Refining requirements, adding functionality, changing scope                   |
| **bug-\*.md**         | Bugs with "knowledge value" (L2 only). Simple bugs → `buglog.md` only (L1)          | Refining bug understanding, adding root cause analysis                        |
| **refactor-\*.md**    | Refactoring intent, goals, approach                                                 | Refining refactoring approach, updating scope                                 |

**How `product-spec.md` Relates to Other Files:**

- **product-spec.md** defines **WHO** uses the system and **HOW** success is measured
- **project-intent.md** defines **WHAT** to build and **WHY** (scope, constraints)
- **feature-\*.md** defines **WHAT** each individual feature does
- **decisions/\*.md** define **HOW** to implement technically

`product-spec.md` bridges the gap between high-level project vision and granular features. It ensures all AI agents share the same understanding of users, priorities, and success criteria — even when working on different features independently.

**Quick Reference:**

| Situation                        | File to Create/Update     | Action                                  |
| -------------------------------- | ------------------------- | --------------------------------------- |
| Define product vision/users/KPIs | `product-spec.md`         | Create or update                        |
| New Feature                      | `feature-*.md`            | Create new file                         |
| Update Feature                   | `feature-*.md` (existing) | Update existing file                    |
| Deprecate Feature                | `feature-*.md` (existing) | Mark as deprecated, keep file           |
| Bug Fix (simple)                 | `buglog.md`               | Add entry (L1, always)                  |
| Bug Fix (complex, L2)            | `bug-*.md` + `buglog.md`  | Create intent if meets Level 2 criteria |
| Resolve Bug                      | `bug-*.md` (existing)     | Mark as resolved, keep file             |
| Change Project Scope             | `project-intent.md`       | Update (only for scope changes)         |
| Technical Decision               | `decisions/*.md`          | Create new file                         |

**Do NOT update `project-intent.md` for**: Adding individual features, fixing bugs, updating existing features, technical decisions.

**Do NOT update `product-spec.md` for**: Individual feature details, technical implementation choices, bug descriptions.

## Decisions, Patterns, and Agents

### Decisions (ADR)

**What**: Technical decisions documented with context, decision, rationale, alternatives, and outcomes.

**When to Create:**

- **Step 1 (Intent) - Recommended**: Plan technical decisions when you know the approach
- **Step 2 (Build) - Flexible**: Create decisions if technical choices emerge during implementation
- **Step 3 (Learn) - Outcomes**: Update decisions with outcomes, create improvement decisions if learnings suggest better approaches

**When to Document:**

- Architectural decisions
- Technology choices
- Design patterns
- Important implementation choices
- Asset management decisions (storage, organization, optimization, external integrations like Figma MCP)

**Format**: Simple markdown with Context, Decision, Rationale, Alternatives, Outcomes.

---

### Patterns & Anti-patterns

**What**: Knowledge that evolves with the system and guides development.

**When Patterns are Established:**

| Step                | Activity                                                                                                  |
| ------------------- | --------------------------------------------------------------------------------------------------------- |
| **Step 1 (Intent)** | For new projects: Define initial patterns. For existing projects: Identify existing patterns              |
| **Step 2 (Build)**  | Use established patterns, avoid known anti-patterns, optionally identify new patterns                     |
| **Step 3 (Learn)**  | Preserve patterns identified during Build, update patterns based on learnings, document new anti-patterns |

**Pattern Documentation:**

- **Pattern**: What it is, when to use it, why it works, examples
- **Anti-pattern**: What to avoid, why it's problematic, what problems it causes

Both include context and link to decisions and learnings.

### Agents (agent-\*.md)

**What**: Simple reusable execution patterns stored in `context/agents/`. They're markdown files that define how to execute specific tasks by referencing Brainiac Context files.

**Key principle**: Agents should **reference** context, not duplicate it. All details are in context files - agents just tell AI how to execute.

**When to create an agent:**

- ✅ You find yourself writing the same detailed prompt repeatedly
- ✅ You want to standardize how certain tasks are done
- ✅ You're working with a team and want shared execution patterns
- ✅ You have a complex task that needs structured steps

**Don't create an agent when:**

- ❌ Simple prompts work fine (just reference context files)
- ❌ The task is one-time only
- ❌ Context files already have all the information needed

**Simple Agent Format:**

```markdown
# Agent: [Agent Name]

## Purpose

[What this agent does - one sentence]

## Context Files to Load

- @context/intent/product-spec.md
- @context/intent/feature-[name].md
- @context/decisions/[number]-[name].md
- @context/knowledge/patterns/[pattern].md

## Execution Steps

1. Load context files listed above
2. [Step 1 - what to do]
3. [Step 2 - what to do]

## Output

- [What this agent produces]
```

**Using agents**: Reference in your prompt: `Execute @context/agents/agent-[name].md for [feature]`.

**Creating agents**: Create manually in `context/agents/` or use the local `create-agent.md` prompt pack adopted by the project.

For advanced agent patterns, see [ADVANCED.md](ADVANCED.md).

### Agent Granularity — When to Split

Agents can be organized at four different granularities, each with clear trade-offs. Choose one primary axis per project; mixing granularities is allowed but must be deliberate (documented in AGENTS.md or project README).

**The four granularity axes:**

| Axis | Pattern | When to prefer | Risk if overused |
|---|---|---|---|
| **Framework step** | `agent-planner.md`, `agent-developer.md`, `agent-reviewer.md`, `agent-devops.md`, `agent-insights.md`, `agent-qa.md` | Always — these are the 6 Standard Agents (see [`agent-templates/`](agent-templates/)), foundational to any project | None — these are the baseline |
| **Technical layer** | `agent-backend.md`, `agent-frontend.md`, `agent-database.md` | Monorepo or layered stack with clear technical boundaries; small team (2-4 devs); consistent tech per layer | Too coarse for large projects — one `agent-backend.md` becomes a catch-all dump |
| **Module / feature** | `agent-auth.md`, `agent-checkout.md`, `agent-realtime.md`, `agent-shopping-mode.md` | Medium-to-large projects with isolated features, each with distinct conventions or stack nuances; phased execution (see [grocery-list-app-a example](examples/grocery-list-app-a/context/agents/)) | Many near-empty agent files if features are trivial; duplication of cross-cutting rules |
| **Domain / bounded context** | `agent-payment.md`, `agent-notifications.md`, `agent-search.md` | Microservices, DDD projects, or features with distinct domain language + specialists | Overlaps with module agents if bounded contexts map 1:1 to modules |

**Decision rules:**

1. **Start with the 6 Standard Agents**. Customize (not replace) using `agent-templates/`. This covers most projects under 10 features.
2. **Add Module/Feature agents** when a single feature has conventions that differ from project defaults — specific libraries, validation rules, or DoD. The grocery-list-app-a pattern (one agent per phase) is a good reference.
3. **Add Technical Layer agents** only if your stack has strong layer boundaries and the team is small. Avoid if you already have Module agents covering the same code — duplication is worse than absence.
4. **Add Domain agents** when bounded contexts don't align with modules — typical in microservices or platforms with multiple product areas.

**Mixing granularities:**

It is allowed and common to mix. Example from grocery-list-app-a:
- 6 Standard Agents (from `agent-templates/`) — always loaded
- 9 Module agents (one per phase) — loaded per feature

The mix works because Module agents **reference** Standard ones instead of duplicating their content. If you find Module agents copying Developer/Reviewer conventions, consolidate into Standard.

**Anti-pattern:** creating an agent file for every file in the codebase. Agents are **execution patterns**, not code maps. If the agent doesn't reduce prompt repetition or standardize a non-obvious workflow, it shouldn't exist.

**Governance:** every `context/agents/agent-*.md` must pass three checks at creation:

1. **Trigger check** — what phrase or task loads this agent? If you can't answer in one sentence, the scope is unclear.
2. **Non-duplication check** — does this agent reference context (intent, decisions, patterns) instead of restating them? If it restates, fix before merging.
3. **Retire check** — under what condition would this agent be deleted? If the answer is "never", it's likely too broad.

Record the granularity choice for the project in `AGENTS.md` under a section like "Agent Granularity Convention" so new team members understand the axis in use.

## AI-Human Collaboration

**Collaboration Pattern**: **Intent** - Human leads, AI assists | **Build** - AI builds, human supervises | **Learn** - AI analyzes, human validates.

**Plan, Approve, Execute Pattern**: Brainiac Context uses this pattern for all AI-assisted work.

1. **Plan** - AI explains what it will do (context creation, code generation, or updates)
2. **Approve** - You review and approve (or request changes), maintain control over execution
3. **Execute** - AI executes only with your approval

**Why it matters**: You maintain control, AI explains before acting, you can refine before execution, reduces rework and improves quality.

**Human Responsibilities**: Lead intent capture and validation, supervise AI execution, review and approve work, validate learnings and insights, make critical decisions.

**AI Responsibilities**: Generate code based on context, suggest solutions and improvements, review code for quality, help identify what needs updating in context, propose context updates.

## Definition of Done (DoD)

Brainiac Context uses **Definition of Done** at the **technical/feature level only** - criteria that every feature implementation must meet before being considered complete.

**Key Points:**

- **DoD is technical only**: Applied during Step 2 (Build) when code is implemented
- **Steps 1 and 3 don't have DoD**: They have flexible "Outputs" instead, as they are more iterative and adaptive
- **Project-level DoD**: Defined by your team/project, not by Brainiac Context. Brainiac Context adds the requirement to update context as part of your DoD
- **When to apply**: Only when implementing features (Step 2), not during planning (Step 1) or learning (Step 3)

**Typical DoD Criteria** (for Step 2 - Build):

- Code implemented and working
- Tests passing
- Code review completed
- **All sensors passed** (see `context/sensors/sensors.md`) — Brainiac Context requirement
- **Context updated** (Brainiac Context requirement)
- **Learn Gate passed** (either context updated, or explicit skip declaration with reason) — Brainiac Context requirement
- Documentation updated
- Deployed (if applicable)

**Acceptance Criteria vs DoD:**

- **Acceptance Criteria** (in `feature-*.md` files) - What the feature needs to do functionally (e.g., "User can login", "Data is saved")
- **Definition of Done** - Process criteria that must be met during implementation (e.g., "Tests passing", "Code reviewed", "Context updated", "Sensors passed")

## Validation by Layer

Every implementation or bug fix should include automated validation appropriate to the layer being changed.

### Principle

Validation should not be generic by default.
The framework does not assume that every change should use the same type of test.

Examples:

- isolated logic may require unit tests
- interfaces and interactive flows may require end-to-end or visual tests
- database changes may require schema, constraint, policy, and regression tests
- integrations may require contract, idempotency, and side-effect validation
- automations and agents may require workflow, tool, and scenario validation

### Minimum obligation after Build

Before a task can be considered complete, it is mandatory to:

1. identify which automated validation is appropriate for the layer being changed
2. execute that validation when it is feasible in the current environment
3. record clearly what was validated
4. if automated validation is not feasible, explicitly declare:
   - the reason
   - the residual risk
   - the alternative verification method that was used

### Quality rule

An implementation must not be considered complete only because code was written.
It should only be considered complete when there is validation evidence compatible with the change that was made.

### Delegation

The framework defines the obligation to validate.
The concrete definition of:

- test type
- tool
- command
- expected evidence

belongs to the project context, the specialist agents in `context/agents/`, and the sensors configuration in `context/sensors/sensors.md` (see next section).

## Sensors

Sensors are the **feedback layer** of the framework. They are the tools that return signal to the model after code is generated, so the model can self-verify and react before declaring a task complete.

### Relationship to Validation by Layer

- **Validation by Layer** = **principle** — "choose the right validation type for the layer being changed"
- **Sensors** = **implementation** — "here are the tools active in this project, and how the model reacts to each signal"

They are complementary, not redundant. Validation by Layer defines the obligation; Sensors defines the execution.

### What Goes in `context/sensors/sensors.md`

Every project has a single `sensors.md` file (created during Init) that lists:

1. **Active sensors** — tool + command for each feedback source
2. **How the model should react** — behavioral rules for each signal type

### `sensors.md` Template

```markdown
# Sensors

## Active Sensors

| Sensor       | Tool        | Command                                 | Signal                      |
|--------------|-------------|-----------------------------------------|-----------------------------|
| Linter       | ESLint      | `pnpm lint`                             | style errors                |
| Type checker | TypeScript  | `pnpm type-check`                       | type errors                 |
| Unit tests   | Vitest      | `pnpm test`                             | contract failures           |
| E2E tests    | Playwright  | `pnpm test:e2e`                         | broken user flows           |
| Review agent | -           | Execute `@context/agents/agent-reviewer.md` | architectural issues    |

## How Model Should React

- **Linter error** → fix before considering task done
- **Type error** → fix before considering task done
- **Unit test failure** → identify if regression (fix) or spec change needed (escalate)
- **E2E failure** → escalate to human before proceeding, do not guess the fix
- **Review agent findings** → address all blockers, document waivers with justification

## Notes

- Sensors run in the Execute phase of Plan → Approve → Execute
- No Build completes without sensors passing, or an explicit "sensor not feasible" justification with residual risk
```

### How Sensors Are Triggered

Sensors are **model-triggered, not human-triggered**. When `/brainiac-context-build` runs:

1. The skill loads `@context/sensors/sensors.md` as part of the Execute phase
2. Before declaring "done", the model runs the commands listed
3. The model reads the outputs and reacts per the reaction rules
4. The result is reported to the human as part of Build completion

The human is not the intermediary between AI and reality — the AI takes ownership of the feedback loop, and the human becomes the reviewer.

### When to Create `sensors.md`

During Step 1 (Intent) / Init: `/brainiac-context-init` creates the file based on the tech stack answered during bootstrap. The file is refined as the project grows (new tools adopted, reaction rules tuned).

## AGENTS.md Integration

Brainiac Context works seamlessly with the **[AGENTS.md](https://agents.md/)** standard, an open format used by over 20,000 open-source projects to guide AI coding agents.

**AGENTS.md** acts as a **router** that directs AI agents to Brainiac Context files. Keep it **succinct** - it should primarily indicate where to find context, not duplicate it:

- **AGENTS.md** = Operational router (setup commands, workflow, references to Brainiac Context files)
- **Brainiac Context** = Strategic context (intent, decisions, knowledge, patterns)

**Together**: AGENTS.md routes AI agents to the right Brainiac Context files, providing both operational guidance and strategic context.

### Operational Router Principle

When a project uses `AGENTS.md`, that file should function as an **operational router** for AI agents.

Its role is to:

- indicate the loading order for relevant context files
- point to the appropriate specialist agents in `context/agents/`
- reinforce project-wide operational rules
- reinforce scope, security, and execution constraints
- guide the expected workflow for the project

### Scope Boundary

`AGENTS.md` should remain concise.

It should **not** become a duplicate copy of:

- intents
- ADRs / decisions
- patterns
- knowledge files

Those artifacts remain the primary source of context. `AGENTS.md` should route agents to them, not replace them.

**Structure**: `AGENTS.md` at project root references `context/` directory. Your `AGENTS.md` should include a "Context Files to Load" section listing relevant context files.

**Important**: AGENTS.md should be **kept updated** to reflect changes in the living context:

- New feature intents → Add to "Feature-Specific Context" section
- New decisions → Add to "Technical Decisions" section
- New patterns → Add to "Knowledge and Patterns" section
- Deprecated features → Update or remove references

### Relationship to Validation

If a project defines validation requirements, `AGENTS.md` may reinforce them in a summarized form.

However, the detailed definition of:

- what kind of validation is required
- which tools should be used
- what evidence is expected

belongs in the project context and specialist agents, not in `AGENTS.md` itself.

**Note**: AGENTS.md is **optional** but **recommended** for better AI agent experience. Brainiac Context works perfectly without it, but together they provide complete guidance.

See [TOOLS.md](TOOLS.md) for more details.

### Canonical vs Project Overrides (since v1.16.0)

Since v1.16.0, `AGENTS.md` is partitioned into two HTML-marker-delimited blocks that separate framework canonical from project-specific content. Since v1.17.0 the `BRAINIAC:CANONICAL-START` marker also carries a `version=X.Y.Z` attribute that is the **source of truth** for the canonical block's version (no longer dependent on regex over visible header text):

```markdown
<!-- BRAINIAC:CANONICAL-START version=1.17.0 | Managed by /brainiac-context-update — DO NOT EDIT manually -->
## Brainiac Context Workflow
## Context Structure
## Context Loading Order
## Code Standards
## Security Standards
## GitHub Operational Protocol
## todo.md vs Project Board (Model A)
## Context Update Protocol
<!-- BRAINIAC:CANONICAL-END -->

<!-- PROJECT:OVERRIDES-START | Editable; preserved across updates -->
## Project Identity
## Project Operational Rules
## Project Artifacts Map
## Project Limits
## Project Agents (Extensions)
## Source of Truth Hierarchy
## Project Environment
<!-- PROJECT:OVERRIDES-END -->
```

**Contract:**

- The **CANONICAL block** is owned by the framework. `/brainiac-context-update` reads `templates/AGENTS.md.template` from the canonical repo and rewrites this block whenever the framework version changes. **Do not edit by hand.** The `version=` attribute on `BRAINIAC:CANONICAL-START` is the source of truth for drift detection — robust to header translations or formatting changes.
- The **PROJECT:OVERRIDES block** is project-owned. Skills `brainiac-context-init`, `brainiac-context-existing`, and `brainiac-context-feature` may populate or extend it; updates never touch it.
- Ordering is **canonical-first** so AI loading `AGENTS.md` reaches framework rules first, then applies project overrides.

**Legacy AGENTS.md** (pre-v1.16, no markers) is detected by `/brainiac-context-update` and migrated via IA archeology: each existing section is classified as canonical (discard, fresh from template), project (preserve in overrides block), or hybrid (human confirms split). The migration is interactive and gated by APPROVE.

**AGENTS.md v1.16.x without `version=` metadata** (pre-v1.17): detected as `UPGRADE_AGENTS_MARKER` (🔵 deterministic, non-breaking). The skill adds the version attribute automatically on the next refresh — no migration prompt needed.

**Companion file**: `.claude/CLAUDE.md` (router specific to Claude Code) imports `AGENTS.md` via `@AGENTS.md` syntax and adds Claude-Code-specific helpers (hooks list, command references). It does not duplicate AGENTS.md content. Generated by `brainiac-context-init` (and refreshed by `brainiac-context-update` when missing or legacy).

**Canonical path** (v1.18+): `.claude/CLAUDE.md`. The Claude Code CLI supports both `./CLAUDE.md` (root) and `./.claude/CLAUDE.md` as project memory paths with **equal precedence** (see [Claude Code memory docs](https://code.claude.com/docs/en/memory.md): "A project CLAUDE.md can be stored in either `./CLAUDE.md` or `./.claude/CLAUDE.md`"). Brainiac chooses `.claude/CLAUDE.md` deliberately so the entire harness — `settings.json`, `hooks/`, `CLAUDE.md`, project-local `install.sh` — lives under one directory, keeping the project root clean and the harness boundary explicit.

**Legacy migration**: projects on pre-v1.18 may have `CLAUDE.md` at the root. `/brainiac-context-update` detects this as `MIGRATE_CLAUDE_MD_PATH` (🔵 deterministic: backup datado + `mv` to `.claude/CLAUDE.md`). Content is preserved integrally — only the path changes.

## Hooks Layout (v1.18+)

The Claude Code hooks directory is partitioned in the same fashion as `context/agents/` — canonical content owned by the framework lives under `brainiac/`, project-specific customization lives under `project/`. This keeps the `apt install` vs `apt upgrade` model consistent across the entire harness.

```
.claude/hooks/
├── brainiac/                        ← Framework canonical (overwritten by install/update)
│   ├── canonical-gate.sh             (PreToolUse — layer 2 of canonical defense)
│   ├── clean-code-gate.sh            (PreToolUse — SEC-1 + Clean Code hard gates)
│   ├── learn-gate.sh                 (Stop — Learn step enforcement)
│   ├── prompt-reminder.sh            (UserPromptSubmit — trigger pattern detection)
│   ├── qa-gate.sh                    (Stop — QA advisory)
│   ├── require-intent.sh             (PreToolUse — Intent-loaded enforcement)
│   ├── session-start.sh              (SessionStart — framework contract injection)
│   └── messages/                     (i18n: en.sh, pt-BR.sh)
└── project/                         ← Project-defined hooks (NEVER touched by install/update)
    └── (user hooks)
```

**Path-based detection rule:**

> The skill `/brainiac-context-update` **only touches** entries in `.claude/settings.json` whose `command` points to `.claude/hooks/brainiac/`. Entries pointing to `.claude/hooks/project/` (or any other path) are considered **user-managed** and **NEVER modified**.

| Path in `command` | Update touches? | Reason |
|---|---|---|
| `.claude/hooks/brainiac/<name>.sh` (v1.18+) | ✅ Yes — overwrites cleanly | Canonical, framework-managed |
| `.claude/hooks/brainiac-<name>.sh` (pre-v1.18 flat) | ✅ Yes — migrates to `brainiac/` | Legacy canonical, same Brainiac |
| `.claude/hooks/project/<name>.sh` | ❌ No — preserved | User-managed |
| `.claude/hooks/<anything-else>.sh` (no `brainiac-` prefix) | ❌ No — preserved | User-managed (legacy) |

**Reserved namespace:** `.claude/hooks/brainiac/` is a **reserved path**. Never place project-specific hooks under that subfolder — the update skill will overwrite them. Always use `.claude/hooks/project/` for custom hooks.

**JSONC markers parking:** JSON strict (RFC 8259) does not support comments, so visual markers like `// BRAINIAC:CANONICAL-START` inside `settings.json` would break the Claude Code parser. The framework currently relies on **path-based detection** alone. When/if Anthropic supports JSONC (see open issues [#29370](https://github.com/anthropics/claude-code/issues/29370), [#17968](https://github.com/anthropics/claude-code/issues/17968), [#12688](https://github.com/anthropics/claude-code/issues/12688)), visual markers can be added as a clarity layer on top of path-based detection.

**Legacy migration**: projects in v1.11.0–v1.17.x use flat layout (`.claude/hooks/brainiac-<name>.sh`). `/brainiac-context-update` detects this as `MIGRATE_HOOKS_LAYOUT` (🔵 deterministic) plus `UPDATE_SETTINGS_HOOK_PATHS` (🔵 deterministic, sed regex). Both have backups and preserve user-managed hooks integrally.

## Drift Detection — 4 Planes / 4 Skills (v1.19+)

Brainiac Context distinguishes **four orthogonal drift planes**, each with its own epistemology and its own dedicated detection skill. This separation follows the same logic that produced `brainiac-context-audit` and `brainiac-context-lint` in v1.14.0 — bundling drift detection across epistemologies into a single skill creates a god-skill that's hard to maintain and pollutes the Diff Report with mixed signals.

| Plane | Skill | What it detects | Epistemology |
|-------|-------|-----------------|--------------|
| **A — Framework drift** | `/brainiac-context-update --diagnose` | Files canonical vs local (`.brainiac-context-framework.md`, hooks, agents, templates, snapshots `code-standards.md` / `security-standards.md`) | File compare (SHA + structural detection F.5/F.6/F.7/F.8) |
| **B — Implementation drift** | `/brainiac-context-audit` | Codebase vs `context/` docs (e.g. feature.md says "endpoint X exists" but code has no route) | Code analysis (grep + AST + Intent ↔ implementation diff) |
| **C — Documentation drift** | `/brainiac-context-lint` | Internal `context/` self-consistency (ADRs referencing non-existent features, patterns without usage, etc.) | Graph analysis over `context/` files |
| **D — GitHub Operational drift** | `/brainiac-context-github --diagnose` | Labels canonical (8 famílias) + `.github/` templates + AGENTS.md PROJECT:OVERRIDES block + branch protection | `gh` CLI + remote API + filesystem |

**Common operating shape:**

- Each skill has **`--diagnose`** (read-only Diff Report) + **default apply mode** (write under Plan → Approve → Execute)
- Each skill produces a Diff Report with the same 4 categories: 🔵 Determinístico / 🟡 IA archeology / 🔴 Reconciliação manual / ⚪ Sem mudança
- Each skill respects the same Visual Vocabulary (`[Drift:🔵 Determinístico]` prefix in stdout)

**Cross-skill signaling.** The `update` skill performs a **light cross-signal check** in Etapa D.7: if `.github/ISSUE_TEMPLATE/` or AGENTS.md "GitHub Operational Protocol" block is missing, it emits an INFO finding (`GITHUB_LAYER_DRIFT_HINT`) pointing the user to `/brainiac-context-github --diagnose`. This is **NOT** full detection — just a pointer. Other cross-signals are parking for v1.20+:

- `update` → `/brainiac-context-audit` (Plane B): when changes in `context/agents/brainiac/` are detected but no matching implementation evidence in codebase (B26.x candidate)
- `update` → `/brainiac-context-lint` (Plane C): when `context/decisions/` has new ADRs without related-files cross-references

**Why this separation matters:**

- **SRP** — each skill has one clear purpose, easier to maintain and extend independently
- **Epistemology coherence** — file SHA compare, code analysis, graph analysis, and `gh` API live in different mental models
- **Graceful degradation** — `update` works offline (no `gh` needed); `github --diagnose` works in `local-only` mode when `gh` is unavailable; `audit`/`lint` depend on file system only
- **Diff Report clarity** — Plane A findings stay focused on framework canonical drift, not polluted by labels diff from GitHub server

## Code Standards as Build Contract

Brainiac Context ships with [`CODE_STANDARDS.md`](CODE_STANDARDS.md) — a versioned, AI-First Clean Code contract that complements [PRINCIPLES.md](PRINCIPLES.md). While PRINCIPLES describes *how the agent works* (the Karpathy Harness, the 3-step cycle), CODE_STANDARDS describes *what the code generated by the agent should look like*.

Together, both documents form the **engineering layer** of Brainiac as an AI-First operating system: the agent works under PRINCIPLES, and the code it produces follows CODE_STANDARDS.

### Universal Layer + Local Snapshot

CODE_STANDARDS operates in two layers:

| Layer | Source | Who decides |
|---|---|---|
| **Universal** | `Brainiac-Context/CODE_STANDARDS.md` (framework canonical) | Framework — non-negotiable (5 Hard gates + ~25 Soft via reviewer) |
| **Local snapshot** | `context/code-standards.md` (per-project) | Project — stack-specific overrides + threshold adjustments via ADR |

The local snapshot is **material** (a copy, not a link), versioned via `Standards-Version: X.Y` header, and pinned to a specific framework version. This makes CODE_STANDARDS a **Build Contract**: each project knows which version of standards it is operating against, and updates are explicit (via `brainiac-context-update`).

### Automatic Loading

Standards are loaded into the agent automatically at 7 mechanisms in the framework, without human invocation:

| Mechanism | Action |
|---|---|
| **Bootstrap** | `brainiac-context-init` copies template → `context/code-standards.md` |
| **Build (Step 2)** | `agent-developer.md` loads `context/code-standards.md` |
| **Real-time hook** | `brainiac-clean-code-gate.sh` enforces deterministic subset of Hard gates (filename banned, file >500 lines, `: any` without justification) on every Write/Edit |
| **QA Gate hook** | `brainiac-qa-gate.sh` enforces bugfix-without-regression-test on Stop |
| **Sensors** | `tsc --noEmit` / `eslint` / `ruff` enforce broken lint/typecheck at end of Build |
| **Quality Gate** | `brainiac-context-learn` Phase 1 invokes `agent-reviewer.md` with standards as checklist (remaining Hard gates + Soft + Info) |
| **Update** | `brainiac-context-update` propagates new framework versions |

Humans never need to remember to load standards. Implicit in the framework, like file permissions in an OS.

### Two Phases of Step 3 (Learn)

With CODE_STANDARDS active, the Learn step splits into two sequential phases:

1. **Phase 1 — Quality Gate**: the reviewer agent classifies findings as 🔴 Block / 🟡 Soft / 🔵 Info, blocks Phase 2 if Block findings exist
2. **Phase 2 — Knowledge Capture**: existing Learn workflow (intent updates, ADR outcomes, patterns/anti-patterns, buglog, changelog)

Block findings cannot be deferred. Soft findings require human decision. Info findings feed Phase 2.

### Out of Scope

CODE_STANDARDS v0.x covers hand-written product code. The following are explicitly out of scope: Markdown, workflow JSON (n8n, Make, Zapier), schema files (OpenAPI, GraphQL SDL), generated code (`*.gen.*`), lockfiles, SQL migrations and dumps, and declarative configs. Each category has different invariants and may receive its own standards document if pain emerges.

### Exemption Mechanism

When a Hard rule objectively applies but the case is genuinely legitimate, an inline marker provides escape:

```ts
// brainiac-allow-large reason="lookup table; splitting fragments domain"
```

The reviewer agent checks every exemption and questions weak justifications. PR reviewers see the marker in the diff.

See [CODE_STANDARDS.md](CODE_STANDARDS.md) for the complete contract.

## Security Standards as Separate Layer

Brainiac treats **security** as its own layer, parallel to (not nested within) CODE_STANDARDS. The two layers serve different threat models:

| Layer | Concern | Failure mode |
|---|---|---|
| **CODE_STANDARDS** | Reader cognitive cost, maintainability, agent retrieval | Bad names cost grep; long files cost context window |
| **SECURITY_STANDARDS** | Confidentiality, integrity, authorization, secret leakage | Leaked token = compromised account; disabled RLS = data breach |

A `wip-` filename produces friction. A leaked `ghp_*` token produces an incident. They cannot share the same Hard gate scheme without trivializing one or both.

### Universal Layer + Local Snapshot

Same pattern as CODE_STANDARDS:

- **Universal**: 3 Security Hard Gates (SEC-1, SEC-2, SEC-3) live in framework root [`SECURITY_STANDARDS.md`](SECURITY_STANDARDS.md), non-negotiable
- **Local**: project snapshot at `context/security-standards.md` with optional stack-specific overrides (additional token patterns, project test/mock whitelist paths)

The snapshot is created by `/brainiac-context-init` and `/brainiac-context-existing` from `templates/security-standards.md.template`. Versioned via `Standards-Version: X.Y`.

### Automatic Loading (3 mechanisms)

| Point | Mechanism | When |
|---|---|---|
| **Real-time hook** | `brainiac-clean-code-gate.sh` blocks known SEC-1 token patterns (`ghp_*`, AWS `AKIA*`, etc.) at write time | On every `Write`/`Edit` |
| **Quality Gate** | `brainiac-context-learn` Phase 1 invokes `agent-reviewer.md` with SECURITY_STANDARDS as security checklist (SEC-1 full + SEC-2 + SEC-3) | At end of every Build phase |
| **Optional sensor** | `gitleaks` / `trufflehog` runs in CI (recommended, not required) | On every push/PR |

### Severity rule

Security findings are **always Block** when detected without valid exemption. There is **no Soft tier and no Info tier for security** — security violations are binary.

### Exemption mechanism

Only **SEC-3** (disabled security primitives) accepts an inline exemption marker:

```ts
// security-allow reason="Public RSS feed endpoint; CSP would block 3rd-party
// reader compatibility; tracked in ADR-007"
res.setHeader('Content-Security-Policy', '');
```

**SEC-1 and SEC-2 have no exemption.** The reviewer agent rejects `// security-allow` markers attached to those gates.

### Cross-reference with CODE_STANDARDS and GitHub layer

- CODE_STANDARDS, PRINCIPLES, SECURITY_STANDARDS form a **triad** of orthogonal contracts (how the agent works / what the code looks like / what must never appear in code)
- GitHub authentication (`integrations/github/PROTOCOL.md §9`) is the operational counterpart of SECURITY_STANDARDS: it documents *how* to avoid leaking tokens in CI/CD setups; SECURITY_STANDARDS documents *what to do when one leaks anyway*

See [SECURITY_STANDARDS.md](SECURITY_STANDARDS.md) for the complete contract.

## GitHub as Operational Layer

Brainiac handles **context, decisions, knowledge, and learning**. GitHub handles **execution, traceability, review, and merge**. The two layers are complementary, not overlapping:

| Layer | Source of truth | Answers |
|---|---|---|
| **Brainiac context** (`context/`) | Why, what was decided, what to follow, what was learned | Intent, ADR, patterns, sensors, evolution |
| **GitHub** | Who is executing, which branch, which PR, evidence attached, merge history | Issues, Labels, Projects, Branches, PRs, Commits, Reviews |

The GitHub Operational Protocol formalizes this integration. When adopted, every relevant change follows:

```
Intent → GitHub Issue → Branch → Build → Draft PR → Sensors/QA → Learn Gate (Phase 1 + Phase 2) → Human Merge
```

### Hard rule

```
Relevant change without an Issue = operational flow violation.
Code without Intent/ADR when needed = Brainiac violation.
PR without evidence = not ready.
Build without Learn Gate = not concluded.
```

### What the protocol provides

The `integrations/github/` directory contains the operational artifacts:

| Artifact | Purpose |
|---|---|
| [`integrations/github/PROTOCOL.md`](integrations/github/PROTOCOL.md) | Full operational specification — flows, label taxonomy, Project structure, branch/commit conventions, agent auth |
| [`integrations/github/templates/.github/ISSUE_TEMPLATE/feature.yml`](integrations/github/templates/.github/ISSUE_TEMPLATE/feature.yml) | Feature issue template (Brainiac Intent fields, AC, validation) |
| [`integrations/github/templates/.github/ISSUE_TEMPLATE/bug.yml`](integrations/github/templates/.github/ISSUE_TEMPLATE/bug.yml) | Bug issue template (L1/L2/L3 triage, evidence, DoD) |
| [`integrations/github/templates/.github/pull_request_template.md`](integrations/github/templates/.github/pull_request_template.md) | PR template (Brainiac context, validation, Phase 1 Quality Gate output, Phase 2 Learn Gate) |
| [`integrations/github/scripts/setup-labels.sh`](integrations/github/scripts/setup-labels.sh) | Idempotent script that creates the official Brainiac label taxonomy (8 families, ~70 labels) |

### Mapping Brainiac Intent → GitHub Issue

| Brainiac artifact | GitHub Issue title pattern |
|---|---|
| `context/intent/feature-*.md` | `feat(scope): implement <feature-name>` |
| `context/intent/bug-*.md` (L2) | `bug(scope): fix <bug> following bug intent` |
| `context/intent/bug-*.md` + `context/decisions/NNN-*.md` (L3) | `bug(scope): fix <bug> with ADR-NNN` |
| `context/intent/refactor-*.md` | `refactor(scope): <objective>` |

For L1 bugs (simple/local), buglog is sufficient if there is no branch/PR. If there is a branch, an issue exists too.

### Label families

The protocol defines 8 official label families (see PROTOCOL.md §5):

```
type:*       — bug, feature, refactor, chore, docs, test, security, infra, decision, research
priority:*   — p0 (critical) → p3 (low)
status:*     — triage, ready, in-progress, blocked, needs-intent, needs-adr, needs-smoke, needs-qa, needs-review, validated, done
area:*       — universal base (brainiac, context, intent, decisions, knowledge, sensors, evolution, qa, agents, docs, github, ci, infra, security) + project-specific extension
risk:*       — low, medium, high, prod-impact, data, security, llm-behavior, migration, integration
source:*     — production, eval, qa, user-feedback, brainiac, manual-test, logs, client, incident
size:*       — xs, s, m, l, xl
brainiac:*   — intent-required, adr-required, learn-gate, bug-l1, bug-l2, bug-l3, sensors-required
```

Project-specific `area:*` labels (e.g., `area:auth`, `area:checkout`, `area:supabase`) are added per project and documented in the project's `AGENTS.md`.

### Adoption

`/brainiac-context-init` will (in v1.12+) run `setup-labels.sh` and copy the templates into the project's `.github/` directory by default, with explicit opt-out. Until that integration ships, projects can adopt manually:

```bash
# Idempotent — safe to run multiple times
bash <BRAINIAC>/integrations/github/scripts/setup-labels.sh OWNER/REPO
cp -r <BRAINIAC>/integrations/github/templates/.github <PROJECT>/.github
```

### Authentication for AI agents

Three options (PROTOCOL.md §9):

1. **GitHub CLI** (`gh auth login`) — recommended for local development
2. **Fine-grained PAT** — for agents/automation; minimum permissions only
3. **GitHub App** — for mature framework deployments (per-repo scope, granular permissions, better audit)

**Hard rule**: never paste a token in a prompt, issue, PR, README, or versioned file.

### `todo.md` vs Project Board (Model A)

Brainiac formalizes two complementary tracking surfaces with **distinct semantics** — never duplicate, never confuse:

| Surface | Purpose | Audience |
|---|---|---|
| **`context/evolution/todo.md`** (local, markdown) | Broad backlog + ideation + speculative B-items (includes "maybe someday" entries) | Solo dev or small team brainstorming offline |
| **GitHub Project Board** (remote) | Only prioritized executable work (Triage → Ready → In Progress → Review → Done) | Team coordination, stakeholder visibility |

**Promotion flow**:

1. Idea / observation lands in `todo.md` (free-form, no friction)
2. When prioritized → becomes GitHub Issue → enters Board at **Triage** column
3. Triage → priority decision → moves to **Ready** (queued) or **Blocked** (external dep)
4. Worked on → **In Progress** → Draft PR → **Review** → **Done** (merged)
5. Item closed on Board → marked `[x]` in `todo.md` OR moved to `context/evolution/changelog.md` if milestone (L3)

**Rationale**:

- Markdown is good for ideation; Boards are good for execution
- Conflating them forces every idea into formal issue creation (high friction for beginners / non-technical users) OR keeps Board cluttered with speculative items (low signal)
- Model A keeps each surface optimized for what it does best
- Aligns with B12 (small-steps enforcement) — users can dump rough ideas in `todo.md` without commitment; agent helps promote priorities to Board with discipline

Full operational details: [`integrations/github/PROTOCOL.md`](integrations/github/PROTOCOL.md) §10 (Instruction for AGENTS.md).

### Cross-reference with Code Standards as Build Contract

The Code Standards layer (universal Hard gates + reviewer agent) and the GitHub Operational Layer are **independent but complementary**:

- Code Standards enforces **what** the code looks like.
- GitHub Protocol enforces **how** the change moves from intent to merge.

The PR template (above) requires Phase 1 Quality Gate output (from `agent-reviewer.md`) before the PR is marked Ready for Review. This is where the two layers meet operationally.

See [`integrations/github/PROTOCOL.md`](integrations/github/PROTOCOL.md) for the complete specification.

## Framework Extensions

Projects can extend Brainiac Context with custom artifacts, as long as the extension **derives from an existing framework hook**. This keeps the line clear between "framework oficial" and "project customization" while allowing real projects to grow beyond the core.

### Rules for Extensions

1. **Derive from a hook**: the extension must be justified by an existing framework mechanism (e.g., "Validation by Layer > Delegation" allows a project to define its own validation artifacts like `context/qa/`).
2. **Document in `context/`**: the project must document the extension in a `Project Extensions` section inside its local copy of the framework spec (or equivalent project document).
3. **Reinforce in `AGENTS.md`**: extensions must be referenced in the project's `AGENTS.md` (under operational rules or "Where to Place Each Artifact").
4. **Do not conflict with framework rules**: extensions add; they never override the core 3 steps, DoD, No Limbo Rule, Learn Gate, or other framework invariants.

### Common Extension: `context/qa/`

Dated validation evidence — immutable records of test runs, audits, security reviews, regression batches, or any structured validation execution.

**Base hook**: `Validation by Layer > Delegation`.

**Format**: `YYYY-MM-DD-HHMM-<slug>.md` (filenames cannot use `:`, so `HHMM` without separator — see Timestamp & Timezone Convention)

**Purpose**: Each file is a snapshot of what was verified, when, by whom, and with what result. Useful for compliance, audit trails, or regression tracking.

**Not to be confused with**:
- `evolution/buglog.md` — bug fix history (L1)
- `knowledge/patterns/*` — reusable rules
- `knowledge/anti-patterns/*` — reusable pitfalls

**Example `qa/2026-04-16-checkout-e2e-run.md`**:
```markdown
# QA Run: Checkout E2E Regression

## Context
Regression suite after `decision 014-payment-provider.md` was applied.

## What Was Validated
- All 23 Playwright E2E flows for checkout
- RLS policies for payment tables
- Webhook idempotency

## Result
22/23 passed. 1 failure: edge case with expired coupon — logged as `bug-expired-coupon-checkout.md` (L2).

## Who / When
- Run by: Kal-EL
- Date: 2026-04-16
- Duration: 14 min
```

### Extension Checklist

Before creating a new artifact type in your project:
- [ ] Which framework hook justifies this extension?
- [ ] Is this documented in `Project Extensions`?
- [ ] Is `AGENTS.md` updated?
- [ ] Does it conflict with any framework invariant? (If yes, redesign.)

## Session Recovery Protocol

Long AI sessions get compacted, summarized, or resumed. When this happens, context is lost and the model risks continuing without the needed grounding.

### Recovery Rules

When a conversation is **compacted, resumed, or a new session is started on an ongoing task**, the agent must first reconstruct:

1. **Files changed** in the current task (git status, modified files)
2. **Relevant intent** (`feature-*.md` or `bug-*.md` being worked on)
3. **Relevant ADRs** (decisions referenced by the intent)
4. **Relevant specialist agent** (the agent for the layer being changed)
5. **Sensors configuration** (`context/sensors/sensors.md`)
6. **Pending `context/` updates** (what should be updated at Learn Gate but hasn't been)

**Only after this reconstruction** may the agent continue execution.

### When Recovery Applies

- Conversation compacted by the runtime
- Session resumed after interruption
- New AI session on an in-progress task
- User explicitly says "resumindo" / "continue" / "retomando"

### What Not to Do

- Do not guess what was being done — verify against `context/`
- Do not skip reconstruction to "save time"
- Do not continue a Build without re-loading the original intent and ADR

Skipping recovery violates the Learn Gate and the Plan → Approve → Execute contract.

## Security by Design

Security in Brainiac Context is built into the framework from the ground up.

**Security Principles:**

1. **Security by Design**: Security considered at every step
2. **Context Security**: Living Context contains sensitive information - requires access controls, encryption for sensitive context, context classification (the `brainiac-context-deliverable` skill codifies the first concrete classification table for artifact → audience routing — see [`skills/brainiac-context-deliverable/SKILL.md`](skills/brainiac-context-deliverable/SKILL.md))
3. **Traceability**: All security-relevant actions must be traceable
4. **Least Privilege**: Access to context and systems follows principle of least privilege
5. **Defense in Depth**: Multiple layers of security protection

**Security in Each Step:**

- **Step 1 (Intent)**: Identify security requirements, document security constraints, define security acceptance criteria
- **Step 2 (Build)**: Implement security controls, follow security best practices, document security decisions
- **Step 3 (Learn)**: Monitor security metrics, review security incidents, update security practices

**Security Best Practices**: Start with security from the beginning, document security decisions, regular reviews, keep context secure, monitor continuously in production.

## Language Convention

All Brainiac artifacts follow a clear separation: **canonical structure in English**; **filled content in the user's language**.

This applies across the framework — ADRs, intent files, buglog entries, code-standards snapshots, GitHub issue bodies, PR descriptions, code-review findings — anywhere humans (or agents) consume meaningful text.

### Why this matters

- **Structure in English** keeps cross-project tooling stable. Reviewer agents (`agent-reviewer.md`), search heuristics, and humans switching between projects expect `## Summary`, `## Acceptance criteria`, `## Risks`, `## Status` to live in known positions. Translating headers breaks discoverability and forces every tool to maintain header dictionaries per locale.
- **Content in the user's language** maximizes communication quality. Forcing a pt-BR team to write PR descriptions in English produces shallow, error-prone text that degrades reviews and downstream learning capture. The team understands its own domain best in its own language.
- **Filenames, labels, commit prefixes, and branch names in English canonical** because they participate in universal conventions (Conventional Commits, GitHub label search, filesystem-safe naming) and cross-team integration.

### Rule by layer

| Layer | Language | Why |
|---|---|---|
| **Template structure** — section headers in `pull_request_template.md`, `bug.yml`, `feature.yml`, ADR template, intent template, buglog template, code-standards template (`## Summary`, `## Validation`, `## Why`, etc.) | **English canonical** | Reviewers (human and agent) locate evidence in consistent positions across projects |
| **Filled content** — body of PR, issue, ADR, intent, buglog entry, code-standards overrides, reviewer findings text | **User's language** | Communication quality; team understands its own domain best |
| **Labels** (`type:bug`, `priority:p0`, `status:ready`) | **English canonical** | Cross-project + GitHub label search compatibility |
| **Commit prefixes** (`feat:`, `fix:`, `docs:`, `chore:`) | **English canonical** | Conventional Commits is a universal standard |
| **Branch names** (`feat/123-cart-validation`) | **English canonical** | URL-safe, cross-team, integrates with CI/CD tooling |
| **Filenames** in `context/` (e.g., `feature-cart-validation.md`, `bug-checkout-stuck.md`, `001-supabase-rls-strategy.md`) | **English canonical** | Cross-project search; avoids accent/encoding issues |
| **Code identifiers** (variable names, function names, class names) | **English canonical** | Universal coding convention; language-tool compatibility |
| **Code comments** | **User's language** (when comments are needed at all — see code-standards §2) | Same reason as filled content; team understanding |
| **README.md, AGENTS.md, CLAUDE.md** of a specific project | **User's language** | Project-facing docs serve the project team |
| **Brainiac framework root docs** (this file, `CODE_STANDARDS.md`, `PRINCIPLES.md`, `PROTOCOL.md`) | **English canonical** + optional `pt-BR/` copies | Framework targets multi-locale adoption; canonical EN enables global discoverability |

### How agents detect user language

Agents resolve the user's language via the following precedence:

1. **`BRAINIAC_LANG` environment variable** — if set to `en`, `pt-BR`, `es`, etc., use it
2. **Existing project content** — read sample of `context/intent/`, `context/evolution/buglog.md`, or recent commit messages; infer dominant language
3. **Conversation context** — language the user is currently writing in
4. **Fallback** — `pt-BR` (current Brainiac default audience)

Agents must apply the detected language consistently within a single artifact (don't mix pt-BR and English in the same PR body).

### Examples

**PR body** (structure EN, content pt-BR — typical Brainiac project):

```markdown
## Summary

Adiciona validação de quantidade mínima no carrinho para evitar checkout
com itens removidos durante a sessão.

## Linked Issues

Closes #42

## Brainiac Context

- Intent: `context/intent/feature-cart-min-quantity.md`
- ADR: `context/decisions/007-cart-validation-strategy.md`

## Context

A sessão do usuário expira mas o estado do carrinho persiste — antes desta
mudança era possível fazer checkout com itens já removidos do catálogo.

## Validation

- [x] Sensors executed (tsc + eslint clean)
- [x] Local tests executed
- [x] Smoke test executed
```

**ADR** (structure EN, content pt-BR):

```markdown
# 007 — Cart validation strategy

## Status

Accepted (2026-05-11)

## Context

Carrinho persiste entre sessões mas catálogo é dinâmico — produtos removidos
ainda aparecem no carrinho até checkout.

## Decision

Validar cada item contra o catálogo no momento do checkout, removendo
silenciosamente itens não disponíveis e mostrando aviso ao usuário.
```

**Buglog entry** (structure EN, content pt-BR):

```markdown
### 2026-05-10 14:32 — Checkout aceita carrinho com itens removidos do catálogo

## What (Symptom)

Usuário consegue completar checkout com produtos que foram removidos do
catálogo antes da finalização — pedido entra com SKU inválido.

## Root Cause

Validação de catálogo só roda no momento de adicionar ao carrinho, não no
checkout.
```

**Issue body** (structure EN, content pt-BR):

```yaml
Observed behavior: |
  Ao completar checkout com produto removido do catálogo, pedido entra com
  SKU inexistente e quebra a etapa de fulfillment.

Expected behavior: |
  Sistema deve detectar itens inválidos no checkout e removê-los do pedido
  com aviso ao usuário.
```

### Out of scope

- **Translating Brainiac framework root docs** to every locale on demand. Only `pt-BR/` exists today as a paired copy of canonical EN files. Other locales are best-effort by the agent when reading, not maintained as canonical translations.
- **Mixing languages within a single artifact**. Pick one and stick to it.
- **Proper nouns and technical terms** (`useEffect`, `Supabase RLS`, `Edge Function`, `K6`, `Vitest`) don't translate regardless of language — keep them as-is.

### Migration

Existing projects with pt-BR content in EN-canonical positions (filenames, labels, branches) or vice-versa: no forced backfill. New artifacts created from v1.12.0 onwards follow this convention. Agents may suggest migration during routine work but should not block.

## Timestamp & Timezone Convention

All dated artifacts (changelog, buglog, intent Status, decision Outcomes, validation logs) use a single global format and timezone.

### Format

**`YYYY-MM-DD HH:MM`** — 24-hour clock, no seconds.

### Timezone

All timestamps refer to **America/Sao_Paulo (UTC-03:00, UTC-02:00 during DST)**.

The timezone is **implicit** — do NOT append `(BRT)` or `(-03:00)` to the timestamp string. The convention is established globally; agents and humans treat all timestamps as Sao Paulo local. This keeps changelog/buglog clean.

### For agents (Claude Code on host, Paperclip in container)

- **Generating a timestamp**: `$(date +'%Y-%m-%d %H:%M')` in shell.
- **Containers (Paperclip)**: `TZ=America/Sao_Paulo` is set at image level (see `integrations/paperclip/docker/Dockerfile`) so `date` returns BRT.
- **Host (Windows/macOS/Linux dev)**: assumes the system clock is set to BRT. Devs in other timezones must override per-command: `TZ=America/Sao_Paulo date +'%Y-%m-%d %H:%M'`.

### Variants for special cases

- **Filenames** (cannot use `:`): `YYYY-MM-DD-HHMM` — ex: `2026-05-03-2045-checkout-e2e.md`
- **Machine-parseable logs** (ISO 8601, when timezone must be explicit): `2026-05-03T20:45:00-03:00`

### Examples

```markdown
## 2026-05-03 20:45 — Phase 1 Setup mergeado em main (HAS-11)

### 2026-05-03 14:32 — PAT GitHub bloqueou push HAS-4

**Created**: 2026-05-03 14:32 (Phase: Intent)
**Updated**: 2026-05-03 20:45 (Phase: Learn) - phase 1 closed

Filename: 2026-05-03-2045-phase1-validation.md
```

### Migration

Entries created **before** this convention (date-only, no time-of-day) remain valid as historical records. New entries from v1.10.0 onwards use the timestamped format. No backfill required.
